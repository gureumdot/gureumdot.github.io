kv cache

算法 笔记
技术笔记
Publish Date:   2026-05-23

KV cache 是推理 infra 的"鼻祖"——几乎所有后续的推理优化(PagedAttention、GQA/MQA、prefix caching、连续批处理……)都建立在它之上。但它本身常常被讲错。本文只抠三个最容易理解错的点:一个大前提 + 两个易错结论。

一、大前提:模型每一步真正需要的,只有"最后一行"

一句话收束:最后一行已经包含了前面的所有信息，大模型做预测用最后一行足以

把一次前向想成一个 [seq_len × d] 的张量穿过四个模块。关键在于:第 i 行的输出,到底依赖哪些行的输入?

为什么只取最后一行就够了?自回归生成的目标是 P(下一个 token | 已有上文)。在因果掩码下,序列里第 4 行(最后一行)是唯一"看过完整上文 token₁…₄"的位置;它经过 attention 聚合后的隐藏态,本质就是"对 token₅ 的预测表示"。然而后续的FFN层token之间相互独立，直到最后的 LM Head 作用在这最后一行上,输出的 logits 即 P(token₅ | token₁..₄)。前面 1~3 行算出来的 logits 对应的是"预测 token₂/₃/₄",在生成阶段早已用过、不再需要。

二、易错点 ①:KV cache 到底优化了"哪一部分"?

一句话收束:省去的计算是kv从输入x经过linear projection的计算，而不是attention的计算

这是全文最关键的一点,也是绝大多数博客讲反的地方。

直接被缓存、被省掉的,是 K 和 V 的 linear projection。

  • 没有 cache 时:每生成一个新 token,你都要把整个序列 X∈Rn×dX \in \mathbb{R}^{n\times d}X∈Rn×d 重新做一遍 K=XWKK = XW_KK=XWK​、V=XWVV = XW_VV=XWV​。前面 n−1n-1n−1 个 token 的 KV 明明上一步刚算过,却被白白扔掉重算。整段生成下来,投影总量是 O(n2)O(n^2)O(n2)。
  • 有 cache 时:历史的 k1..kn−1,v1..vn−1k_1..k_{n-1}, v_1..v_{n-1}k1​..kn−1​,v1​..vn−1​ 直接躺在 cache 里。每步只对最新一个 token 做投影 kn=xnWKk_n = x_n W_Kkn​=xn​WK​(xn∈R1×dx_n \in \mathbb{R}^{1\times d}xn​∈R1×d),再 concat 进 cache。整段投影总量降到 O(n)O(n)O(n)。

那"attention 计算被优化"这种说法错在哪?它不是错,而是因果搞反了。attention 那一步(qnq_nqn​ 与全部 K 做点积、再加权 V)在 decode 阶段确实从 O(n2)O(n^2)O(n2) 降到了每步 O(n)O(n)O(n)——但这个节省的直接来源是"只需最后一行 q",而不是 KV cache。哪怕你完全不缓存 KV、每步重算所有 KV,只要你只拿最后一行的 qnq_nqn​ 去做 attention,这一步本来就是 O(n)O(n)O(n) 的。

三、易错点 ②:为什么没有 Q cache?

一句话收束:Q 不进 cache,是因为大前提我们只需要最后一行，因为其他模块的计算token相互独立，所以attention只需要最后一行，然而attention最后一行的计算只需要最新的 q。所以我们不需要前面的q来计算前面的attention行

很多人会问"既然缓存 K、V,为什么不缓存 Q?"——这个问题本身就隐含了一个误区:它把"丢掉历史 q"当成 KV cache 的特性,其实两者无关。

回到 attention 第 4 行的计算:

out4=softmax ⁣(q4⋅[k1,k2,k3,k4]⊤dk)⋅[v1,v2,v3,v4]\text{out}_4 = \text{softmax}\!\left(\frac{q_4 \cdot [k_1, k_2, k_3, k_4]^\top}{\sqrt{d_k}}\right) \cdot [v_1, v_2, v_3, v_4] out4​=softmax(dk​​q4​⋅[k1​,k2​,k3​,k4​]⊤​)⋅[v1​,v2​,v3​,v4​]

观察这个式子里谁出现了:

  • K、V 用到了全部历史(k1..k4k_1..k_4k1​..k4​、v1..v4v_1..v_4v1​..v4​ 全要)——所以它们必须留下来,这才有了 cache。
  • Q 只用到了 q4q_4q4​ 一个(q1,q2,q3q_1, q_2, q_3q1​,q2​,q3​ 在这个式子里根本不出现)——历史的 q 算完即弃,根本没有"留"的必要,自然也就没有 cache 的概念。

关键区分: "丢掉历史 q"是**「只需最后一行」的直接推论**,在第一节就已经成立了——它不依赖你是否做 KV 缓存。换句话说,把"没有 Q cache"和"有 KV cache"并列成一个对偶问题是个陷阱:前者是输出结构决定的(只要最后一行),后者是为了省投影计算而引入的工程优化。两者的根在不同地方。

四、易错点 ③:"丢历史 q"只在 decode 成立,prefill 绝不能丢

一句话收束:能不能扔历史 q,只取决于一件事——那些行的 K/V 是否已经算完存在了。

  • decode:历史 K/V 早已算完躺在 cache 里,新一步只需新 token 的 q 去产新 K/V → 历史 q 可丢。
  • prefill:历史行的 K/V 还没算完(这趟正要算它们),而算它们要用到这些行的 q,且下一层还要吃 → q 一个都不能丢。

问题的核心在于:每一层的输出就是下一层的输入 x,而算 KV 需要完整的 x。

假设你在 prefill 阶段，输入序列有 4 个 token。有人会这样想：“最终我只要最后一行的输出，那中间行 1/2/3 的计算是浪费，每层只保留第 4 行传下去不就好了？” 这个直觉在单层里是对的，但 Transformer 是多层堆叠的，这个做法在第二层就直接崩了。

根本原因只有一句话:每层的输入 x 是上一层的完整输出,算 KV 要用到 x 的每一行。一旦中途丢掉行 1/2/3,下一层就没有 x₁, x₂, x₃,对应的 KV 根本无从计算。

decode 之所以能"只用最后一行",是因为历史 token 所有层的 KV 在 prefill 阶段已经全部算完存进了 cache——decode 每步进来的 x 本来就只有一行(新 token),不存在"丢不丢"的问题。两者的区别不是"要不要丢行",而是 prefill 时 KV 还不存在、decode 时 KV 已在 cache 里。

附:一个常被追问的延伸——KV cache 是怎么变成"显存杀手"的

面试里如果你把上面讲清楚,面试官很可能顺势追问"那它的代价是什么"。简短答案:KV cache 把计算换成了显存。每个 token、每层、每个 head 都要存一份 K 和一份 V,显存占用大致是:

KV cache 显存≈2×nlayers×ntokens×nkv_heads×dhead×dtype_bytes\text{KV cache 显存} \approx 2 \times n_{\text{layers}} \times n_{\text{tokens}} \times n_{\text{kv\_heads}} \times d_{\text{head}} \times \text{dtype\_bytes} KV cache 显存≈2×nlayers​×ntokens​×nkv_heads​×dhead​×dtype_bytes

(系数 2 来自 K 和 V 各一份。)它随序列长度线性增长,长上下文场景下很快就吃满显存——这正是后来 MQA / GQA(减少 n_kv_heads)、PagedAttention(像虚拟内存一样分页管理 KV)、KV 量化等一系列工作的出发点。这条线索可以单独再开一篇,这里先埋个引子。
