RLHF 推理优化 训练优化
技术笔记
Publish Date:   2026-05-27

一、RLHF 三阶段：verl 优化的对象

verl 的所有优化，都是围绕 RLHF（以 PPO/GRPO 为例）的三个阶段展开的。一个训练 step 的骨架：

for prompts in dataloader:
    # Stage 1: response 生成（rollout）
    batch = actor.generate_sequences(prompts)

    # Stage 2: 训练数据准备
    batch = critic.compute_values(batch)
    batch = reference.compute_log_prob(batch)
    batch = reward.compute_reward(batch)
    batch = compute_advantages(batch)

    # Stage 3: actor 和 critic 训练
    critic_metrics = critic.update_critic(batch)
    actor_metrics  = actor.update_actor(batch)

先把这三阶段的计算性质钉死，后面所有优化的"为什么"都从这里推出来。

Stage 名字 在干什么 计算性质 bound
1 Rollout（生成） actor 自回归生成 response decode（逐 token 串行，复用 KV） memory-bound
2 训练数据准备 算 old_logprob / ref_logprob / reward / value prefill（喂完整 seq，一次 forward） compute-bound
3 模型训练 actor / critic 的 forward + backward + update prefill（+ backward，重复 ppo_epochs 次） compute-bound

💡 三阶段的本质对立：stage1 是 decode，stage2/3 是 prefill。 这是 verl 一切设计的根。

  • stage1 rollout：第 t 个 token 依赖第 t-1 个的输出，必须串行、每步只算一个新 token、复用前面的 KV → batch 维度小、算力喂不饱 → memory-bound。
  • stage2/3：(prompt+response) 整条 seq 在 stage1 就已知，forward 时所有 token 并行进去算大矩阵乘，无自回归依赖 → compute-bound。

易错点：stage3 “多轮训练” ≠ decode 性质

stage3 有"一次 exp 多轮训练"（PPO 的 ppo_epochs：同一批 rollout 数据重复喂、做多次梯度更新）。别把"多轮"误判成 decode。

两个正交维度，别混：
  "算一条 seq 的方式":  并行(prefill) vs 自回归(decode)  ← 决定 compute/memory bound
  "这个动作做几次":     一次 vs 多次(ppo_epochs)         ← 和 bound 无关
  • decode 的"下一步"输入依赖"上一步"的输出（生成的 token）——数据依赖的串行。
  • ppo_epochs 的"下一轮"输入还是那条原始 seq，只是模型参数变了——同一份数据多次利用，没有 token 级串行。

判据：看"单次 forward 内 token 是并行算还是自回归串行"，而不是"做几次"。 重复多次 prefill 不会让单次 prefill 变成 decode。所以 stage3 每一次 forward 都是 prefill 性质、compute-bound，只是多了 backward + 重复若干次。

二、Stage1 优化：Rollout

Rollout 的优化分两条主线，两条都和"引入 vLLM 当推理后端"绑定：

  1. 引入 vLLM（2.1 / 2.2 共同的大前提）——把 rollout 从"直接 actor.generate()" 换成 vLLM 引擎。
  2. 魔改 vLLM 以适配 RL 场景——vLLM 原本为"独占整卡的 serving 服务"设计，塞进 RL 后水土不服，要改一堆。按训练后端不同分两条路径：
    • 2.1 vLLM + FSDP
    • 2.2 vLLM + Megatron

2.0 为什么 rollout 要引入 vLLM：vLLM 帮在哪

本节只列 list、点到为止。vLLM 各机制的细节见另一篇《vLLM》笔记 → [PagedAttention / Prefix Caching]。

vLLM 机制 对 rollout 有没有用 说明
PagedAttention / KV cache 管理 ✅ 最大头 动态分配代替静态预分配，省显存 + 消碎片。RLHF 一个 batch 内 prompt 长度方差极大，静态预分配 max_len 浪费巨量显存 → 分页后能塞更大 rollout batch。
高效算子（FlashAttention、sampling kernel） ✅ 纯加速。
Prefix caching ✅（GRPO 场景天然命中） parallel sampling（一个 prompt 采 n 个回答，GRPO 就这么干）时，n 个 sample 共享同一段 prompt 的 KV，只存一份。
Continuous batching ❌ rollout 不是推理服务，没有"请求陆续到达、动态塞进来"的场景——一批 prompt 进去、一次性生成完整批交给 stage2/3。动态进出的价值用不上。
Chunked prefill ❌ 它解决 prefill 卡 decode 的混合调度，rollout 不混（一步做完进 stage2/3，不会有新 prefill 进来）。

💡 一句话：rollout 加 vLLM，本质收益 = KV cache 管理（省显存）+ 快算子 + GRPO 下 prefix caching 自然命中。 continuous batching / chunked prefill 这俩 serving 的招在 rollout 用不上。

🔗 埋钩子（异步 RL）：现在有一类异步 RL（rollout 与 train 解耦、各跑各的），但那需要 rollout engine 和 train engine 各持一份权重（拿显存换时间）。verl 经典的 HybridEngine 是共享卡、串行的，所以不异步。留待后面展开。

2.1 vLLM + FSDP

大前提：verl 用 HybridEngine 让训练和推理跑在同一批 GPU 上（不是分开两套卡）。所以 vLLM 这个 serving 引擎被塞进 RL 后，几个"独占整卡"的默认假设全失效，要逐个魔改。下面每个魔改点都对应"一个原假设在 RL 下失效"。

2.1.0 [占位] Ray → SPMD

🚧 待补充。 vLLM 原本自带一套基于 Ray 的多卡调度（单独 server 进程），verl 要把它改造成 SPMD 模式、移除 Ray 依赖、保持 torchrun 方式，才能塞进 verl 的训练框架统一调度。这块较大且复杂，先留位，以后单独补。

2.1.1 魔改 gpu_memory_utilization：从"占整卡"改成"占剩余"

原生 vLLM 的设计与隐含假设：vLLM 启动时一次性吃掉 total × util 的显存圈成地盘，内部分两块：

total × util  这块地盘  =  模型权重+激活 (dummy_usage，试跑一次 dummy input 测出的峰值)
                        +  KV cache 配额 (剩下的全给 KV)

剩下的 total × (1 − util) 留作安全缓冲——serving 跑起来显存会临时波动（长序列激活变大、NCCL/CUDA context、碎片），util=1.0 吃满会 OOM，留 10% 当缓冲垫才稳。

隐含假设：整张卡只有 vLLM 一个住户。 util 含义很自然——“我占整卡的 90%，留 10% 给自己波动”。

RL 下假设崩了：同一张卡还住着 actor / critic / ref / reward。这时"剩下的空间"角色变了——它不再是 vLLM 的私有缓冲，而要分给训练模型。于是 util（占整卡比例）这个旋钮失效了：你真正想控制的是"vLLM 在扣掉自己权重后、剩下的可分配空间里吃多少"，但这隔着一个"其它模型占了多少"的变量。换并行配置（TP/PP/DP）→ 其它模型显存变 → 得手算 vLLM 还剩多少 → 反推 util → 试错。这正是博主那句痛点：

“否则需要时刻关注不同并行情况下其它部分的显存占用，反复调整 gpu_memory_utilization。”

verl 的改法：把 util 重定义成剩余显存的比例——先扣掉 vLLM 自己的 dummy_usage，再在剩下的里按 util 切。

分配策略 原生 vLLM verl 改进版
总可用显存 total × util total − dummy_usage
KV cache 配额 total × util − dummy_usage (total − dummy_usage) × util

举例（80G 卡，dummy_usage=20G，util=0.5）：原生给 KV 80×0.5−20=20G；verl 给 (80−20)×0.5=30G。

💡 本质：改的不是数值，是把基准从"整卡"换成"扣掉自己权重后的剩余"。util 语义从"占整卡"变成"占我自己的自留地"，和别的模型占多少解耦——设 util=0.5 永远是"自留地的一半给 KV"，不管邻居怎么变并行，含义不变，不用反复手调。这是适配"多模型共享一张卡"的关键。

2.1.2 [占位] 显存生命周期：offload + free_cache_engine

🚧 待补充/待确认。 这块涉及几个"用时间换显存"的动作，需要和你进一步对齐后再写准：

  • offload_model_weights：rollout 前后把 vLLM 权重在 GPU↔CPU 间挪（"指向 CPU empty tensor + empty_cache 回收"的语义，以及 vLLM 权重究竟何时在 GPU、何时在 CPU 的完整潮汐周期，待确认）。
  • free_cache_engine：generate 结束即释放 vLLM 预分配的 KV cache engine，下次再 build。本质时间换显存，默认开启，收益大。

共同目的：rollout 结束后让 GPU 腾出完整大块显存交给 stage2/3。先留白。

2.1.3 sync_model_weights = resharding（核心）

sync 和 resharding 是一回事。 从 actor 把权重灌进 vLLM 这个"灌"的动作，本身就要做布局转换——因为 actor 的布局（FSDP）和 vLLM 的布局（TP）不一样。布局相同，sync 只是 copy；布局不同，sync 的核心难点就是 resharding。所以：resharding 是 sync 的最大问题，sync 是 resharding 的发生场景。

resharding 要处理两样东西的转换：权重（布局不同）+ 数据（DP 并行度不同）。

本质矛盾：训练布局 ≠ 推理布局

同一份逻辑权重（如某个 q_proj），在训练侧和推理侧被切成不同的物理碎片、分布在不同 GPU 上。resharding 就是把"按训练方式切碎、散在各卡的权重"，重新组装、再按"推理方式切碎、散到各卡"。

以博主的配置为例：world_size=8，vllm tp_size=4，FULL_SHARD（ZeRO-3）。

训练侧：           推理侧：
8 卡 FSDP 分片  →  8 卡 = tp_size(4) × dp_size(2)
(每卡存 1/8,        TP 组0: GPU0-3  → 4 卡合存 1 份完整模型,按 TP 切
 无语义结构)        TP 组1: GPU4-7  → 另 4 卡存另 1 份完整模型,按 TP 切
                    两组之间是 DP(各跑不同 rollout 数据)

注意：训练侧 8 卡全参与切分（每卡 1/8），推理侧 tp=4。8 和 4 对不上，中间差的因子 2 就是 vLLM 侧的 dp。

关键易错点：FSDP（FULL_SHARD）的切法不是 TP，是"压平后平分"

这是整条线最容易讲错、也最值得讲透的点。

先分清 FSDP 计算时 ≠ PP/TP：FSDP 每卡存 1/8 的全部参数（压平的），但计算时先 all-gather 把权重凑回完整，然后所有卡各自用完整权重算不同的 data——本质是 DP。压平分片只是存储手段，不是计算手段。它既不是 TP（不是每层切片协作算同一份数据）也不是 PP（不是按层切流水）。

为什么 FSDP 非要压平平分，而不像 TP 那样按 weight 切？ 两个原因，后者才是根本：

  • （顺带好处）通用性：传统 TP 有结构约束（按 head 切要整除 head 数、按 dim 切要整除 hidden）。压平后纯按字节平分，任何 shape 都能切 N 份。
  • （根本原因）集合通信的字节绝对均衡：FSDP 的核心操作是 all-gather 凑权重 / reduce-scatter 散梯度，是集合通信，耗时由最慢的卡决定（木桶效应）。要无短板，每卡持有的字节数必须严格相等。而传统 TP 的"按语义切"做不到严格相等——layernorm 这种一维小张量不切、每卡复制一份，加上不规则 shape，各卡字节必然有差异。要严格相等，唯一干净的办法就是：无视 layer / 语义 / shape，把所有参数首尾拼成一根一维长向量，按总字节 ÷ N 一刀切。

📌 [图：TP 按 weight 各切片 vs FSDP 压平平分] — 已绘制，对比两种切法。重点：FSDP 的切口（按字节平分的边界）与原 weight 边界完全不对齐，所以 GPU0 拿到的是"完整 q_proj + 半截 k_proj"这种横跨多个 weight、无语义的碎块。

🔗 埋钩子（TP 字节不均衡）：传统 TP 因小张量复制 / 不规则 shape，各卡字节并非严格均衡——FSDP 压平正是为消除这种不均衡。细节待展开。

后果：FSDP 必须 full_tensor() 还原，Megatron 不用

vLLM 的 TP 组（GPU0-3）要的是"完整的 q_proj，再按 TP 规则切 4 份"。但 q_proj 被压平后碎成几段、散在全部 8 张卡上（不只 0-3，因为压平是跨全部 rank 平分的）。于是：

  • FSDP 路径：必须先 full_tensor()——把全部 8 卡的分片 all-gather 拼回完整一维向量、reshape 还原成 q_proj 等真 tensor，这一步在显存里产生一份完整权重的峰值占用，然后才能按 vLLM 的 TP 重新切（走 ColumnParallel / RowParallel 的 weight_loader，按 tp_rank narrow 出自己那段）。sync 完立刻 del + empty_cache() 释放这份临时完整权重。
  • Megatron 路径：切分本来就是结构化的（q_proj 按 head/dim 规则切，和 vLLM TP 同构），能更直接地对应过去，不需要"先还原成完整权重"这个 all-gather 峰值。（细节见 2.2。）

💡 这就是你最早那个易错点的精确版：“FSDP 要多存一份 vLLM 参数、Megatron 不用” 的本质是——FSDP reshard 必须先 all-gather 出完整权重（产生峰值副本），Megatron 因切分与 vLLM TP 同构可省掉这一步。

weight_loader：name 对齐 + layout 对齐

把 actor 权重灌进 vLLM 不只是"切对分片"，还要解决 actor model 和 vLLM 内部模型实现对不上的问题。verl 对每个模型架构 hard-code 一个 loader（llama_dtensor_weight_loader / deepseekv2_dtensor_weight_loader 等），吃掉两层差异：

  • name 对齐：vLLM 的参数命名规则和 HF 开源模型不同，要查表翻译（如 layers.0.qkv ↔ vLLM 内部名）。
  • layout 对齐（更麻烦）：vLLM 为 kernel 效率会把权重 concat。最典型 QKV——HF 里 q/k/v 常是三个分开的权重，vLLM 内部合并成一个 qkv_proj。MoE 的 FFN 也有 concat vs separate 的差异。loader 要处理"三个 tensor 怎么拼成 vLLM 要的那一个"。

一句话：对齐有两层——name 对齐（改名字）+ layout 对齐（qkv 该 concat 还是 separate、moe 怎么排），后者才是真麻烦。

数据侧 resharding：8 份 → 2 份 → 8 份

权重布局转换的同时，数据也要 reshard，因为并行度对不上：非 vLLM 组件是 dp=8（8 卡各吃不同数据），vLLM 是 dp2×tp4（只有 2 份完整模型）。

进 generate 前 (preprocess)：
  GPU0-3 各自 bs=4 的数据  ──concat──>  data0 (bs=16),喂 TP 组0
  GPU4-7 各自 bs=4 的数据  ──concat──>  data1 (bs=16),喂 TP 组1
  (TP 组内 4 个 rank 必须吃【相同】输入 → 打包成一份;
   用 allgather_dict_tensors 沿 dim=0/batch 维聚合)

generate 后 (postprocess)：
  bs=16 结果  ──split──>  切回 4 个 bs=4,发回各 rank
  (恢复 dp=8 布局,后续 stage 各 rank 又吃不同数据)

💡 为什么 TP 组内必须吃相同输入：TP 是把一份模型切到 4 卡上协作算同一份数据。所以同一 TP 组的 4 个 rank 必须吃完全相同的 batch，否则 4 张卡的分片对不上、算出来的东西无意义。这是 TP “协作算一份” 语义的直接推论。

2.1 小结：相比 actor.generate() 改了什么

维度 改动
架构解耦 分离 actor model 与 rollout model，vLLM 当推理后端，吃它的快算子 + KV cache 管理
框架适配 vLLM 改造成 SPMD、移除 Ray、保持 torchrun（[占位 2.1.0]）
显存配额 gpu_memory_utilization 重定义为"占剩余"（2.1.1）
显存生命周期 offload + free_cache_engine，用时间换显存（[占位 2.1.2]）
参数同步 sync = resharding：FSDP full_tensor 还原 → weight_loader 按 tp_rank 分区加载（2.1.3）
数据流 DP-TP 混合并行下数据 concat/split 重打包（2.1.3 数据侧）

2.2 vLLM + Megatron

与 2.1 同级，是 stage1 的另一条 resharding 路径。和 FSDP 的核心区别：FSDP 是"DP + 伪 TP"（压平平分，不同构于 vLLM TP），Megatron 的切分更丰富、且本身就结构化、与 vLLM TP 同构。这带来两个新问题，也带来一个新优化空间。

2.2.1 前置事实：推理侧的并行策略 ≠ 训练侧

训练学的那套并行（TP/PP/DP/EP）是为训练设计的；搬到推理，不是全都能用：

并行方式 vLLM 推理是否用 原因
TP ✅ 主力 推理切分基本靠 TP。
DP ✅ 多份完整模型各跑不同数据。
PP ❌ 不用 pipeline bubble 大 + 通信复杂；decode 每步算量极小，bubble 占比被放大到无法接受，microbatch 流水在自回归 decode 下也难填满。
EP（MoE） ✅（MoE 专属） dense 模型用不到，先留钩子，本节按 dense 讲。

💡 结论：vLLM 推理侧只有 TP + DP（dense 模型）。 训练侧 Megatron 用了 PP，推理侧 PP 维度消失——这就是下面 “PP→DP” 转换的来源。

2.2.2 Megatron 比 FSDP 多出来的转换：两个维度叠加

训练侧 (Megatron, stage2/3 共用):  TP4 × DP2 × PP(若干)   ← prefill 性质,compute-bound
            ↓ resharding (两个变化叠加)
推理侧 (vLLM, stage1 rollout):     TP2 × DP4 (无 PP)       ← 主要服务 decode,memory-bound

变化一：PP 维度消失 → PP→DP（不是 PP→TP）

PP 是按层切的（pp_rank=0 存前 N 层、pp_rank=1 存中间 N 层……每个 pp stage 是模型的一段连续 layer，不是完整模型）。而 vLLM 一个 TP 组要的是一份完整模型（所有层）。

为什么走 PP→DP 而不是 PP→TP？因为一条完整的 pp 流水线（pp_rank 0→1→2 串起来）天然就是一份完整模型的所有层——把这一整条流水线当成 DP 的一路即可，TP 维度可保持不变，几乎是自然转换。而 PP→TP 要把"按层纵切"打破、改成"按矩阵横切"（PP 切的是层归属/模型深度，TP 切的是矩阵内部分片/矩阵宽度，两者正交），是昂贵的重排。所以走 PP→DP 最省。




变化二：TP/DP 配比改变（TP4×DP2 → TP2×DP4）

同样是 TP+DP，但训练（prefill/compute-bound）和推理（decode/memory-bound）的瓶颈不同，最优配比就不同：

  • prefill（compute-bound）：计算是瓶颈，TP 开大（TP=4），每卡分到的计算量小、算得快；all-reduce 通信变多，但被远大于它的计算时间掩盖，划算。
  • decode（memory-bound）：计算本就不是瓶颈，开大 TP 省计算没用，反而每步都要的 all-reduce 通信无法被掩盖、暴露成新瓶颈。所以 decode 偏好小 TP + 大 DP：小 TP 省通信，大 DP 增大有效 batch、把权重搬运成本摊薄（算术强度↑、更接近 compute-bound）。

注意：rollout 内部 prefill+decode 都有，但 vLLM 整个 rollout 用同一套配比（不在 generate 中途 reshard，否则要多转两次）。这套配比向 decode 倾斜，因为 rollout 耗时绝大部分在 decode（prefill 只是 prompt 那一下，decode 要生成几百上千 token）。

🔗 prefill/decode 的瓶颈差异、roofline 推导留到专门的《Prefill vs Decode》章展开，这里只点到为止。

2.2.3 核心优化：micro-DP 组（HybridFlow 的精髓）

这是 Megatron 路径相比 FSDP “省” 的真正来源——但要破除一个常见误述：不是"Megatron 天生省"，而是 HybridFlow 这个设计省；而这个设计能成立，前提恰恰是 Megatron 的 TP 切分与 vLLM TP 同构。

朴素做法（HybridFlow-V）：训练侧每个 rank 持权重的一个 TP 分片，转推理时做一次全 TP 组 all-gather 凑出完整权重，再丢弃不属于自己的部分。这一步 all-gather 出完整权重，产生完整权重峰值显存——和 FSDP 的 full_tensor() 一个性质。

优化做法（HybridFlow）：verl 重新设计 rank 分组规则（MICRO_DATA_PARALLEL_GROUP），让 TP<DP，使组内两个 rank 持有互补分片。这样只需在 micro-DP 小组内 gather，凑出推理需要的那部分即可，不必凑完整权重。


例：tp=2 时分片排布让 rank0/1 互补、rank2/3 互补 → rank0/1 组内 gather 凑出 [m_0,m_1]，rank2/3 组内 gather 凑出 [m_2,m_3]。通信量减半，且不产生完整权重峰值。

⚠️ 易错点（作者也常讲错）：这个优化不是"改切分优先级"就能解释的。

  • 光改优先级，只改变 rank 的编号排布（谁和谁组 DP 组：默认 {0,2}/{1,3} → 改后 {0,1}/{2,3}），不改变"该 gather 完整权重的还是得 gather 完整权重"的本质，解释不了通信减半。
  • 真正的创新是 micro-DP 组：把 gather 范围从"全 TP 组（凑完整权重）“缩小到"micro-DP 小组（凑互补的那部分）”。改优先级只是让"该互补的 rank 排进同一小组"的前提之一，不是全部——还要配上 micro-DP 组的 gather 逻辑（MICRO_DATA_PARALLEL_GROUP，而非 vLLM/Megatron 默认 group）。

2.2.4 FSDP vs Megatron resharding 最终对照

切分性质 与 vLLM TP reshard 怎么凑完整 完整权重峰值
FSDP (FULL_SHARD) 压平按字节平分（伪 TP） 不同构 必须 full_tensor() 全量 all-gather 还原 有（绕不开）
Megatron 朴素 (HybridFlow-V) 结构化 TP/PP 同构 全 TP 组 all-gather 凑完整再丢弃 有
Megatron 优化 (HybridFlow) 结构化 TP/PP 同构 micro-DP 组内 gather 凑互补部分 无，通信减半

💡 “FSDP 多存一份、Megatron 不用” 的精确版：不是"Megatron 不 gather"，而是——FSDP 压平后不同构，没法做小组互补，只能全量还原出完整权重（峰值）；Megatron 同构，HybridFlow 才能把 gather 缩到 micro-DP 组内、不凑完整权重，从而避开峰值、通信减半。省的根因是"切分同构"使小组互补成为可能。

Megatron 路径易错点速查 List

# 易错点 正确理解
1 推理和训练的并行策略一样？ 不一样。vLLM 推理（dense）只用 TP+DP，不用 PP（bubble 大 + decode 算量小放大 bubble）。EP 是 MoE 专属。
2 Megatron→vLLM 为什么是 PP→DP 不是 PP→TP？ 一条完整 pp 流水线天然=一份完整模型所有层，当成 DP 一路即可（TP 不变），几乎免费；PP→TP 要打破层归属重新横切，昂贵。
3 TP4×DP2→TP2×DP4 发生在哪？ stage 之间（训练↔推理），不是 rollout 内部 prefill/decode 之间（否则要多转两次）。配比变是因 prefill/decode 瓶颈不同。
4 Megatron 比 FSDP 省，是 Megatron 天生省？ 不是。是 HybridFlow（micro-DP 组）设计省；前提是 Megatron 切分与 vLLM TP 同构，分片才能小组互补。FSDP 不同构故省不了。
5 HybridFlow 的优化=改切分优先级？ 不是。改优先级只改 rank 编号排布，不改"要不要 gather 完整权重"。真正创新是 micro-DP 组把 gather 缩到组内、凑互补部分（通信减半）。
6 朴素 Megatron 路径没有完整权重峰值？ 有。朴素 HybridFlow-V 全组 all-gather 凑完整权重再丢弃，和 FSDP full_tensor 一个性质。只有优化版（组内 gather）才避开峰值。

FSDP 路径易错点速查 List

# 易错点 正确理解
1 stage3 多轮训练是 decode 性质？ 不是。多轮(ppo_epochs)只是重复 prefill-forward；判据看"单次 forward 内 token 并行还是自回归"，stage3 是 prefill、compute-bound。
2 rollout 加 vLLM 用到了 continuous batching / chunked prefill？ 没用上。rollout 不是推理服务，无动态进出 / prefill-decode 混合。收益是 KV cache 管理 + 快算子 + GRPO 下 prefix caching。
3 gpu_memory_utilization 改的是数值？ 不是。改的是基准：从"占整卡"换成"占扣掉自己权重后的剩余"，让旋钮在多住户场景下语义稳定、不随邻居变。
4 FSDP(FULL_SHARD) 的切分是 TP？ 不是。是"所有参数压平成一维向量后按字节平分"，分片横跨多个 weight、无语义。TP 是每个 weight 各自切片。
5 FSDP 压平平分是为了通用性？ 通用性只是顺带好处。根本原因是集合通信(all-gather/reduce-scatter)要字节绝对均衡(木桶效应)，TP 按语义切做不到(小张量复制/不规则 shape)。
6 FSDP 压平后变成了 PP / 异步计算？ 不是。压平只是存储布局；计算时仍 all-gather 凑完整权重做 DP。既非 PP 也非 TP。
7 reshard 时 FSDP 能直接从某几张卡捞出完整 q_proj？ 不能。q_proj 碎片散在全部 8 卡，必须 full_tensor() all-gather 还原（产生完整权重峰值显存）才能用。
8 "FSDP 多存一份、Megatron 不用"指什么？ FSDP reshard 必须先 all-gather 出完整权重峰值副本；Megatron 切分与 vLLM TP 同构，可省掉这步还原。
9 weight_loader 只是改名字？ 不止。两层：name 对齐(改名) + layout 对齐(qkv concat vs separate、moe 排布)，后者更麻烦。
10 TP 组内各 rank 可以吃不同输入？ 不能。TP 是协作算同一份数据，组内必须吃相同 batch（所以数据要 concat 成一份），否则分片对不上。

三、Stage2/3 优化：数据准备 + 训练

stage2（compute_log_prob / compute_values / compute_reward）和 stage3（update_actor / update_critic）的 forward 都是 prefill 性质、compute-bound，且共用很多优化，合在一起讲。

3.1 为什么 stage2 用训练逻辑，而不是 vLLM

唯一硬约束：数值自洽。 stage2 算的 old_logprob / ref_logprob / value 是 stage3 loss 的输入，PPO/GRPO 的 importance ratio = exp(new_logprob − old_logprob) 对数值差异极其敏感。若 stage2 用 vLLM、stage3 用训练框架，两套实现的 logprob 有数值 diff → 训练崩 / reward hacking。用同一框架、同一套权重布局算 stage2/3 才数值一致。

（"计算模式相同"只是"可以这么做"的前提；"省一次 resharding"不成立——怎么都要转一次——故都不是独立理由。）

易错点：verl 强制在 stage2 重算 logprob，不用 generate 给的

generate 其实能返回每个 token 的 prob，但 verl 强制重算。两个具体原因（即"数值自洽"的细化）：

  1. 后处理精度：generate 时若用了 sampling 后处理（repetition penalty 等），返回的 logits 是被策略改过的值，不是模型原始输出。
  2. 算子 diff：vLLM 算子优化激进，同参数算出的 logits 和 actor model 有数值 diff。这个 diff 可能不改变 candidate 排序（不影响采样结果），但一旦 logits 数值进入后续 loss 计算，就给本就不稳的 RL 加了不确定性。

💡 重算的代价：多花一次 inference 时间"保平安"。而时间大头在 generate（decode 几百上千 token），这点 prefill 重算不心疼。

3.2 Remove padding：让真 token 全程不和 pad 同框

3.2.1 pad 从哪来：为了把变长序列堆成规整张量

单条序列天然是规整矩阵 [seq_len, d_model]，不用 pad。pad 的必要性出现在"多条序列堆成一个 batch"那一步。

GPU 要高吞吐，得一次算一个 batch = 一个三维张量 [batch_size, seq_len, d_model]。要把多条序列沿 batch 维 stack 成规整立方块，中间 seq_len 这一维对整个 batch 只能有一个值。但各序列长度不一（100 vs 4），堆不齐 → 短的补 pad 垫到 max_len。

✗ 堆不齐 (seq_len 不等)        ✓ pad 对齐后能堆
[100, d]                       [100, d]
[100, d]            pad 到100   [100, d]
[4,   d]            ────────→   [100, d] (4真+96pad)
→ 不是规整立方块                → [3, 100, d] 规整 ✓

为什么非要规整张量？ 三层递进（最关键是第二层）：

层面 内容 决定
物理（寻址） 规整 → 统一 stride（b×(S×d) 里 S 是固定常数）→ GPU 能按公式索引每条序列 能不能算
性能（并行）★ 规整 → 整块走同一条指令流（SIMT），所有线程齐步并行、GPU 打满；不规整 → 各序列循环边界不同 → 线程发散（warp divergence）→ 并行度暴跌 算得快不快
现象 torch.stack 不等长直接报错 上两层的表现

💡 pad 的终极目的：喂饱 GPU 并行算力 → 打包成大 batch 齐步算 → batch 必须规整 → 变长序列对齐 seq_len → 短的补 pad。 规整只是手段，并行才是目的。

3.2.2 pad token 是"先全算再丢弃"，不是"被跳过"

matmul 是稠密运算，不认识谁真谁假——它看到 100 行就把 100 行全算了，包括 96 行 pad。我们事后用 attention mask 把 pad 位置的结果遮掉，但计算已经发生，mask 只能丢弃结果、救不回算力。

→ 既然 mask 救不回算力，那就从一开始别让 pad 进矩阵乘 = remove padding。

3.2.3 怎么做到：分层处理

把所有真 token 首尾拼成一根长条 [total_tokens, d]（零 pad），然后分层看：

  • FFN / LayerNorm / projection 等：对每个 token 独立计算（out[i] 只看 x[i]），token 之间不耦合 → "长条 vs 方阵"对它们毫无区别，直接照算，pad 消失。
  • Attention（唯一例外）：是唯一让 token 互相耦合的层（i 要 attend j）。拼成长条后危险——seq0 的 token 会去 attend seq1 的 token（串味）。

Attention 的特殊处理：flash_attn_varlen_func + cu_seqlens

把分段信息从"张量形状（靠 pad 体现的 S 维）"转移到 cu_seqlens 索引表里。cu_seqlens = 各样本实际长度的累积和，标记长条里每段的边界：

bs=2，长度 3 和 4，拼成 7 个 token：[t0 t1 t2 | t3 t4 t5 t6]
cu_seqlens = [0, 3, 7]
              ↑  ↑  └ seq1 结束
              │  └─── seq0 结束 / seq1 开始
              └────── 起点
相邻两数框定一段：[0,3)=seq0，[3,7)=seq1

flash_attn_varlen_func 拿到这张表，只在段内 attend、绝不跨边界 → 串味解决，且全程无 pad、无需 mask 遮 pad。

💡 全程无 pad：输入拼成长条 → FFN/LN 不需要方阵 → attention 用 varlen 也不需要方阵 → 输出。整个 forward 保持无 pad 长条，pad 陪算的浪费被彻底省掉。

3.2.4 是 RLHF 专属优化吗？

不是。 flash-attn varlen + cu_seqlens 是处理变长序列的通用手段，预训练 / SFT 早就在用（sequence packing）。但收益大小 ∝ batch 内长度方差：

remove padding 是通用训练优化，机制非 RLHF 发明。但 RLHF 的 rollout response 长度方差极大（短的几十、长 CoT 上千），pad 占比远高于普通训练，加上 attention 是 O(S²)（一条长4的序列 pad 到 1500，attention 算量从 4² 涨到 1500²），所以 remove padding 在 RLHF 收益被放大，几乎必备。

3.3 大 logits 的显存问题（反直觉）

compute_log_prob 的输出 logits：

logits: [total_tokens, vocab_size]   ← 与模型大小【无关】，只看 token 数 × 词表

vocab_size 是固定大数（约 15 万），不管模型 0.5B 还是 70B 词表一样大。例：bs=16、平均长 1000、vocab=15万、fp32 → 16000×150000×4B ≈ 9.6G，一个中间张量就近 10G。

⚠️ 反直觉现象：“训 0.5B 模型也 OOM。” 因为瓶颈不在模型权重而在 logits 中间激活：小模型省了权重显存，大家就把 batch 开更大 → total_tokens 涨 → logits 更大 → OOM。

老写法的坑（trl）：

logp = F.log_softmax(logits, dim=-1)   # 又造一个 [total_tokens, vocab] 大张量 → 双倍峰值！
logpy = gather_from_labels(logp, labels)

解法：selective log-softmax。利用 log_softmax(x)_i = x_i − logsumexp(x)，logsumexp 把 vocab 维归约成标量（每 token 一个数）→ 中间张量从 [total_tokens, vocab] 降到 [total_tokens]。

💡 本质：我们只需要 label 那 1 个 token 的 log-prob，不必先算出整个 vocab 的 log-softmax 再 gather 一个丢弃。 verl 用 flash-attn 的 cross_entropy 或 logsumexp 公式实现。

3.4 dynamic_bsz：按 token 切，一招解决负载不均 + OOM

📌 [图：按条数切（静态 bsz）vs 按 token 切（dynamic bsz）] — 已绘制。木桶效应 + OOM vs 均衡封顶，附带 selective log-softmax 对比。

remove padding 带来新问题：序列长度方差大时，按"条数"切分导致 token 数严重不均。

按条数切（每组 4 条）：
  rank0: [1500,1200,1400,1300]=5400 tokens  ← 累死
  rank3: [12,30,8,45]=95 tokens             ← 闲死

两个层面的后果：

  1. 跨 rank（DP）→ 木桶效应：DP all-reduce 同步梯度，必须所有 rank 都算完才能 reduce。rank3 算完干等 rank0 → GPU 空转。
  2. 单 rank 内 micro-batch → OOM：某 micro-batch 恰好全长序列 → token 爆 → 大 logits → OOM。

解法：切分标准从"条数"换成"token 数"。

固定条数（静态）：  每 micro-batch = 4 条        → token 不可控,可能爆
固定 token（dynamic）：每 micro-batch = max_token_len → 条数浮动,token 封顶

每组凑够 max_token_len（即 ppo_max_token_len_per_gpu）个 token 成一组，条数因此浮动（全长序列装几条、全短序列装几百条）——这就是 use_dynamic_bsz 里 “dynamic” 的由来。token 封顶 → logits 大小封顶（不 OOM）；各组 token 数相近 → 负载均衡。一招解决两个问题。

Stage2/3 易错点速查 List

# 易错点 正确理解
1 stage2 为什么不用 vLLM？ 唯一硬约束数值自洽：logprob 要和 stage3 用同框架同布局算，否则 importance ratio 数值 diff → 训练崩。
2 stage2 能直接用 generate 返回的 prob？ 不能，强制重算。generate 的 logits 被 sampling 后处理改过 + vLLM 算子有 diff，数值不可靠。
3 单条变长序列要 pad？ 不要。单条天然是规整矩阵。pad 只在多条堆成 batch 时为对齐 seq_len 维才需要。
4 要规整张量只是为了能寻址？ 寻址是门槛（能不能算），最关键是并行（SIMT 齐步算、GPU 打满）；不规整会线程发散、并行暴跌。
5 pad token 会被 GPU 跳过不算？ 不会。matmul 不认识真假，先全算，再用 mask 遮结果。mask 救不回算力 → 故要 remove padding。
6 remove padding 后所有层都要特殊处理？ 不是。FFN/LN 等 token 独立的层直接照算；只有 attention（token 耦合）要 varlen + cu_seqlens 防跨序列 attend。
7 remove padding 是 RLHF 专属？ 不是。通用变长优化（预训练/SFT 都用）。但 RLHF 长度方差极大 + attention O(S²)，收益被放大。
8 大 logits 显存 ∝ 模型大小？ 无关。logits = [tokens, vocab]，只看 token 数 × 词表。小模型开大 batch 照样 OOM。
9 老 log_softmax 写法的问题？ 造一个和 logits 同大的中间张量 → 双倍峰值。selective（logsumexp 归约 vocab 维）省掉。
10 dynamic_bsz 的 “dynamic” 指什么？ 指条数浮动：按 token 数（max_token_len）切而非条数 → token 封顶不 OOM + 各 rank 均衡。
