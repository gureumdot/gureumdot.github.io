一、PagedAttention

本节只解决一个问题：显存利用率，从而能在同样的卡上塞下更多并发请求，把吞吐顶上去。

1. 问题：传统 KV cache 的显存浪费

传统方案给每个请求按 max_seq_len 预分配一整块连续显存。模型支持 2048，哪怕实际只生成 100 个 token，剩下的空间也被占着。论文实测，有效显存利用率经常只有 20%~40%。

浪费分两类（这两个术语严格说不是一回事，面试会抠）：

  • internal fragmentation（内部碎片）：预留了但最终没用上的空间。属于这个请求，但浪费了。
  • external fragmentation（外部碎片）：请求之间的空隙。空闲总量够，但都是不连续的小块，凑不出一段连续大空间给新请求。

2. 解法总纲：两种碎片，两种武器

PagedAttention 把 KV cache 切成固定大小的 block（每个 block 存固定数量 token 的 KV），用一张 block table 把"逻辑连续的序列"映射到"物理分散的 block"。

由此对症下药，两种碎片各有各的解法（这个因果对应是本章核心，别说反）：

深色碎片 (internal)  ←──  动态按需分配（不再预留 max_seq_len）
灰色碎片 (external)  ←──  等大 block + block table 映射（不再要求物理连续）

2.1 动态分配 → 消灭深色碎片（internal）

旧方案的问题：一次性预分配 prompt + max_response 的连续显存。其中 prompt 部分是已知的，decode 部分（response）是预留但还没用上的——哪怕最终只生成了 100 个 token，max_response 对应的整段空间也从一开始就被占着不放。这段"预留但未用"的空间就是 internal frag 的根源。

PagedAttention 的改法：把序列切成固定大小的 block，每次只预分配一个 block，用完了再分配下一个。decode 阶段不需要一次性预留整段 response 空间——当前 block 写满，才去申请下一个，写不满就停在那里。

序列长度 = 100 token，block_size = 16
需要 ceil(100/16) = 7 个 block，按需逐个分配
最后一个 block 装 100 - 96 = 4 个 token，浪费 12 个 token 的空间
全局碎片率 → 最多一个 block 以内，论文里 < 4%

额外好处：显存能在请求间流动复用。别的请求算完释放了 block，正在 decode 的请求就能继续申请——不用像以前那样死死预留一大块等着自己用。

2.2 映射表 → 消灭灰色碎片（external）

灰色碎片的根源是"非连续"。GPU 显存里有大量空闲，但这些空闲是零散的——每块都太小，没有任何一块能容纳一个完整的变长 KV。传统做法要求 KV 在物理上连续，所以那些零散的空闲永远拼不起来，只能闲置——这就是 external frag。

映射表的解法：把逻辑视角和物理视角分开。

KV 的逻辑序列：block 0 → block 1 → block 2 → ...（对请求来说是连续的）
                    ↓           ↓           ↓
物理显存：     地址 A     地址 F     地址 B    （实际散落在任意位置）

线程通过映射表（block table）把逻辑 block 编号翻译成物理地址，然后直接跳过去读写。物理内存不需要连续，线程也能正确访问——映射表赋予了线程"访问非连续内存"的能力。

因此：

  • 那些原本因为"不够大、拼不连续"而闲置的零散空闲，现在可以逐个分配给不同的 block
  • 只要显存里有空闲 block，就能继续装 KV，不存在"有空间却用不上"的灰色碎片

💡 类比 OS 分页：操作系统用页表让进程看到"连续"的虚拟地址空间，物理页框可以散落在 DRAM 各处。PagedAttention 做的是完全一样的事，只不过对象换成了 GPU 显存里的 KV cache。

3. 易错点

3.1 映射表与 coalescing 的关系（最易讲错）

最大的误解

“KV 散落在物理各处，attention 计算岂不是要满显存随机捞数据？warp 里 32 个线程访问的是非连续内存，还能合并成一次内存事务吗？”

这是一个伪命题，一句话就能破：

映射表和内存事务（coalescing）是两个层级、毫不相干的东西。

  • 映射表只做一件事：赋予线程"跨地址访问"的能力（逻辑 block → 物理地址）。
  • 内存事务能不能合并，是 warp / block 调度层面的 kernel 设计问题——你只要在写 kernel 时保证"一个 warp 在一个物理 block 内连续访问"，就能 coalesce。这跟有没有映射表、逻辑连不连续，统统没关系。

换句话说：coalescing 只看一条访存指令里 warp 内线程算出来的物理地址连不连续。映射表只负责"把地址算出来"，算完之后连不连续、能不能合并，是 kernel 循环结构自己保证的事。

关键：warp 不会横跨 block 边界

这正是 kernel 设计层面要保证的事——vLLM 的 paged attention kernel 从不让一个 warp 在一条指令里横跨两个物理 block。kernel 的循环结构是：

for logical_blk in sequence:                    # 一次迭代只处理一个 block
    phys_blk = block_table[logical_blk]          # 查表，标量操作
    base = kv_cache_ptr + phys_blk * block_stride  # 算这个 block 的物理基址
    # ↓ 这条访存指令：warp 内线程只在【这一个】物理 block 内部读
    load KV[base + thread_offset]                # 物理连续 → coalesce ✓
# 处理完这个 block 才进入下一轮，查表换 base，再读下一个 block

所以"非连续"被吸收进了 base 的计算里：跨 block 不是靠"一条横跨的访存指令"过去的，而是靠"循环换一轮 + 重新查表算 base"过去的。同一条访存指令内部永远物理连续，coalescing 一点没丢。

代价只是多了"查表 + 算 base"这些标量指令——而 decode 阶段是 memory-bound，算力大把富余，这点标量开销等于免费。

映射表到底解决了什么——破除图的误导

内容：vLLM 论文 Figure 7（两个请求 A、B 的 KV 同时存储，逻辑 block → 物理 block 散落各处）。
一句话看图结论：逻辑连续、物理分散，靠 block table 定位。

这张图（以及几乎所有讲 PagedAttention 的图）有一个隐含误导：它容易让人以为"PagedAttention 把非连续访问变成了连续访问"。不是的。

真相是：

写 kernel 默认就是连续寻址——读数据就是 首地址 base + i × stride 按顺序读，这是最自然的逻辑（你不用多线程、单纯读个数组也是这样）。

想读非连续的物理地址，就必须额外记录下一个块的首地址 base——而 block table 干的就是这件事：记录每个 block 的物理首地址。

所以 PagedAttention 不是"把非连续变连续"，而是"给了 kernel 寻址非连续物理内存的能力"。

类比（数组 vs 索引表，你写 kernel 一定熟）：

无表（公式寻址）：base(i) = start + i*16*stride    ← 像数组 arr[i]，元素必须连续摆放
有表（查表寻址）：base(i) = block_table[i] * stride ← 像加了索引表，元素可散落任意位置

"必须连续"不是物理限制，是寻址方式的限制。 你选了"用公式推地址"，公式就只能表达连续关系；想打破连续，就得用一张表把任意对应关系记下来。这正是 OS page table 做的事——虚拟页号查表得物理页号，于是物理页能随便放。

正交关系表

很多人把"能不能 coalesce"和"block 能不能不连续摆放"耦合在一起，才会觉得矛盾。其实是两件正交的事：

方案 能 coalesce？ block 能任意摆放（无碎片）？
连续分配，无表 能 不能（必须连续，有碎片）
Paged，有表 能 能
任意分配 + 让 warp 横跨 block 不能 ——

结论：有没有表，不改变"能不能合并"——合并永远靠"每条指令内部物理连续"（由 kernel 循环结构保证）。表只改变"能不能不连续摆放"。映射表从来不是 coalescing 的开关。

3.2 block_size 怎么选

block_size 是个多目标权衡，往两个极端走各有各的死法：

coalescing / 访存 internal 碎片 table 开销 / 灵活性
block_size 小（如 1） 差（每读 1 token 就查表跳走） 小 表长、迭代多
block_size 大（如 256） 好（连续读得多） 大（最后一块浪费多） 表短、但浪费大

核心矛盾：碎片 ↔ 访存效率互相打架（小块碎片小但访存差，大块反之）。vLLM 默认 block_size = 16，是这条矛盾曲线上的经验甜点，且可调（支持 8/16/32）。

⚠️ 重要易错点（必记）：block_size 的选择 不是 为了"对齐 warp=32"或"凑满一次访存事务"。
现代 kernel 是向量化访存（一个线程一次 float4 读 4 个数），访存效率由向量化宽度决定，与 block_size 解耦——16 个 token 的 KV 几个线程就读完了，根本喂不饱、也不需要对齐一个 warp。
block_size 真正权衡的是 { internal 碎片 ↔ block table 开销 ↔ prefix caching 共享粒度 ↔ kernel occupancy }。面试时若有人用"16 对齐 warp"来解释，是错的。 诚实的答法是"16 是经验默认值，不是从硬件带宽推导出的理论最优"。

🔗 埋钩子：block_size 还会影响 prefix caching 的共享粒度——两个请求只有在 block 对齐的前缀上才能共享 KV，block 越大越不灵活。留到 Prefix Caching 章展开。

3.3 Copy-on-Write（CoW）

CoW 自己很少被直接考，但它是 Prefix Caching 的机制底座。本章讲透原理，复用部分留到后面。

触发条件（破第一个误解）

❓ 单序列 decode（一个 prompt → 一个输出）会触发 CoW 吗？
不会。

CoW 只在"一个 prompt 衍生出多个序列"时触发，两种场景：

  • parallel sampling：一个 prompt 采样 n>1 个不同回答
  • beam search：维护 k 条候选 beam

共同特征：多个序列共享同一段 prompt 的 KV。有共享，才谈得上"写时复制"。

机制（用 Figure 7 的 prompt 举例）

parallel sampling，n=2，prompt 占满 Block0 + 半个 Block1：

① 共享阶段：A、B 的 prompt 完全相同，KV 也相同，不存两份，block table 都指向同一批物理 block，每个物理 block 记 refcount。

seq A table: [物理7][物理1] ┐
                            ├─► 物理 block 7、1 共享，refcount = 2
seq B table: [物理7][物理1] ┘

② 分叉 + 写时复制：A、B 采样出不同的下一个 token，都要写进半满且共享的 Block1。若直接写会污染对方的数据，于是 A 触发 CoW：

A 要写 refcount>1 的 Block1
  → 申请新物理 block（物理2）
  → 拷贝原 Block1 已有内容过去
  → A 的 table 改指物理2，在副本上写
  → 原物理1 的 refcount 减到 1（只剩 B 用）
  → B 之后写时发现 refcount=1，独占，直接写，不拷贝

结果：
  seq A: [物理7][物理2(副本)]   物理2 refcount=1
  seq B: [物理7][物理1(原块)]   物理1 refcount=1
  物理7(完整 prompt 的 Block0): 始终 refcount=2, 只读, 永不拷贝 ✓

易错点：哪个 block 会触发 CoW？

不是所有共享 block，只有"半满且还要继续写"的那个。

  • Block0（满的、之后只读）：refcount 一直是 2，永远不拷贝。
  • Block1（半满、还要追加写）：触发 CoW。

💡 关键词是 CoW 里的 “Write”：只有还要往里写的 block 才需要拷贝；满的、只读的 block 可以无限共享、永不拷贝。

🔗 埋钩子（连接 Prefix Caching）："满的、只读的 block 可以无限共享"正是 Prefix Caching 复用的东西——CoW 是"请求内"多序列共享，Prefix Caching 是"跨请求"前缀共享，底层是同一套 refcount + block 复用机制。留到 Prefix Caching 章展开。

3.4 显存打满了怎么办：抢占 / Preemption

问题根源（动态分配的反噬）：动态分配不给任何请求保证"一定有显存跑到结束"。decode 每生成一段就要新增 block，万一某一刻所有运行中的请求都要新 block，而空闲 block = 0 → 爆了。此时只能抢占：牺牲一部分请求，腾出它们的 KV block 给其他请求，被牺牲的请求之后再恢复。

调度原则：FCFS + 后来者先被抢占（LIFO）

  • FCFS（先来先服务）：保证公平，早来的不会被饿死。
  • 如需抢占，后来的先被踢（LIFO）。

💡 为什么踢后来的？ 早来的请求已经攒了一大堆 KV，踢掉它（无论换出还是重算）代价都大；晚来的几乎没攒 KV，踢掉最便宜。

两种回收方式：swap vs recompute

被踢请求已算出的 KV 要腾出来，但恢复时还要用。KV 是"算出来的中间结果"，于是有两条根本不同的路：

swap：      KV 不丢，从 GPU 显存换出到 CPU 内存，恢复时再搬回来
recompute： KV 直接丢掉，恢复时把它当成新 prompt 重跑一遍 prefill 算出来

代价归属（核心，面试爱抠）：

方式 代价花在哪 随序列长度
swap KV 走 PCIe 在 GPU↔CPU 来回搬（PCIe 是系统最窄瓶颈） 代价 ∝ 长度（搬运量大）
recompute 重跑 prefill，吃算力（prefill 是 compute-bound） 代价 ∝ 长度，但单价低（prefill 高并行 + 抢占时算力本就闲置）

📌 图片占位
文件名建议：preemption-swap-vs-recompute.png
内容：左右对比图——swap（KV 经 PCIe 换出 CPU 内存再换回，标注 PCIe 瓶颈）vs recompute（KV 丢弃，恢复时重跑 prefill，标注吃算力）。
一句话看图结论：swap 省算力费 PCIe 带宽，recompute 省带宽费算力。

经典取舍（V0 语境）：两条曲线一交叉——

  • prompt 长 → recompute：序列越长 swap 搬运量越大、PCIe 越慢；而重算吃的是抢占时闲置的算力，prefill 又高度并行，划算。
  • prompt 短 → swap：KV 少搬得快；recompute 有固定 kernel launch / 调度开销，短序列时这笔固定成本占比反而不划算。

⚠️ V0 里 swap 的另一个真正用途：多序列场景（beam search / parallel sampling）recompute 不支持，只能退回 swap 兜底。

V0 → V1 的现状（一句话）

V1 默认直接 recompute，swap 基本退役——因为 V1 默认开了 chunked prefill，被抢占请求恢复时要重跑的那段 prefill，能被切成小块混进正在跑的 decode batch，不会再"重算一大段 prompt 卡住其他请求的 decode"。所以 V1 不再纠结长短，重算开销已被压得很低。

🔗 chunked prefill 到底怎么切、为什么能解决卡顿，留到 Chunked Prefill 章展开。

易错点速查 List

# 易错点 正确理解
1 internal frag 和 external frag 是一回事？ 不是。前者=预留没用上（属于该请求），后者=请求间凑不出连续空间。“预留"严格说也不等于"碎片”。
2 两种碎片靠什么解决？ 深色(internal)←动态分配；灰色(external)←等大 block + 映射表。别说反，更别把"动态分配"算成消灭 external 的根本机制。
3 等大 block 可有可无？ 不可。等大是"消灭 external frag"和"映射表退化成 O(1) 数组"的共同前提（分页 vs 分段）。
4 PagedAttention 把非连续访问变连续了？ 没有。它给了 kernel 寻址非连续物理内存的能力（靠 table 记录每个 block 的 base），访问本身仍是块内连续。
5 block 散落各处会破坏 coalescing？ 不会。warp 不横跨 block 边界，每条访存指令只落在一个物理 block 内（块内连续）。跨 block 靠循环换 base，不靠横跨指令。
6 有映射表才能 coalesce？ 错。coalescing 只看"指令内物理是否连续"，与表无关。表只决定"能不能不连续摆放"。两件事正交。
7 block_size=16 是为对齐 warp=32？ 错。向量化访存下访存宽度与 block_size 解耦。16 是 {碎片↔表开销↔共享粒度↔occupancy} 的经验默认值，可调，非硬件最优。
8 单序列 decode 会触发 CoW？ 不会。CoW 只在 parallel sampling / beam search 这类"一 prompt 多序列共享 prompt KV"时触发。
9 共享的 block 都要 CoW？ 不是。只有"半满且还要写"的 block 触发；满的、只读的 block 无限共享、永不拷贝。

二、Prefix Caching

一句话定位：跨请求复用相同前缀的 KV。它是 CoW（请求内多序列共享）的"跨请求版"，底层是同一套 refcount + block 复用机制。

强依赖 PagedAttention：prefill 复用要靠 block 切分（命中率才高），decode 注册要靠 block-level hash（复用粒度 = block 粒度）。没有 paged 的 block 抽象，prefix caching 无从谈起。

1. 总纲：识别 + 复用 + 省了什么

场景：两个请求独立到达，但前缀相同（最典型的是同一段 system prompt “You are a helpful assistant…”）。它们一开始各算各的，物理 block 不是同一批。Prefix Caching 要让后到的请求复用前人已经算好的前缀 KV。

相对 CoW 多出来的唯一新东西：怎么识别可复用前缀？

靠 hash。维护一张 hash → 物理 block 的映射表（cached_blocks）：

  • hash 算的是 token，不是 KV——所以 prefill 前（KV 还没算）就能算出来。
  • hash 必须是累积前缀 hash（含从头到本块的所有 token，如图里的 hash(A~D)、hash(A~H)），因为一段 KV 取决于它前面所有 token。
  • hash 命中 ⟺ 这段 token 及其所有前缀完全相同 ⟺ KV 必然相同（KV 是 token + 位置的确定性函数）。所以 hash 一致就保证"那个物理 block 里的 KV 正是我这段 token 该算出来的 KV"，无需真去比对 KV。

💡 复用是 block 粒度、从头连续命中：

  • block 切分让复用从"整段全或无"变成"逐 block 命中到哪算哪"，复用率大幅提升——这是 PagedAttention 给的红利。
  • 但必须从头连续命中到第一个 miss 为止：不能跳过中间 miss 的 block 去复用后面的，因为后面 block 的 KV 依赖前面 block 的内容。第一个 miss 之后，即使 hash 偶然相同也不能用。

省了什么（prefill 和 decode 省的位置不同，别混）：

阶段 角色 省什么
prefill 命中 消费者 省重算前缀的 prefill 计算（TTFT↓）+ 不用重存 KV
decode 生产者 几乎不命中、不省 FLOPs；作用是把满 block 注册进 cache，省的是未来请求的显存

⚠️ 常见错误：“prefix caching 只省显存不省 FLOPs”——这是把 decode 的结论错误推广到 prefill。prefill 命中是实打实省计算的，这是它降 TTFT 的核心。

2. Prefill 阶段：查表前置，命中跳过计算（消费者）

要不要建物理块的逻辑：

请求进来（token 已知，KV 未算）
  → 对 token 序列逐 block 算累积 hash（纯 token 哈希，不需要 KV）
  → 拿 hash 查 cache：
       命中 → 直接复用已有物理 block，这段 KV 根本不重算（省计算）
       miss → 从第一个 miss 的 block 开始，才真正 forward 计算、建新块

关键：顺序是"查表前置、命中跳过"，不是"先全算完再发现能复用然后丢弃"。 若真要全算完，prefill 的 FLOPs 一点没省，prefix caching 在 prefill 就失去了最大意义。

前缀单调性（面试爱抠）：复用必须从头连续。hash 累积了从头到本块的所有 token，所以"前 3 块命中、第 4 块 miss"之后就停——第 4 块之后即便 hash 偶然撞上也不能复用，因为它的 KV 依赖第 4 块的内容。

💡 跟"同步/异步"无关：不要用"同步 prefill 能不能做 prefix cache"来想。真正的轴是**“那段前缀的 block 是否已被前人算完、写满、注册进 cache”**。谁先把前缀算熟入 cache，后来者才能蹭。命中只看 block 在不在 cache，与请求何时进 batch 无直接因果。

3. Decode 阶段：写满才注册 hash（生产者）

和 prefill 的根本不同：token 是生成出来才知道的，不是输入已知的。 所以：

  • decode 一个 step 生成一个 token，生成前无法算 hash（不知道 token 是什么）。
  • 只有当一个 block 被逐 token 填满，才对它算一次累积 hash（block-level），注册进 cached_blocks。

要不要建物理块的逻辑：decode 每生成一个 token 写入当前 block，当前 block 写满 → 算 hash 注册 → 申请下一个 block 继续写。

为什么 decode 不省 FLOPs（精确原因）：

  • 不是因为"先算完才查所以省不了"。
  • 而是 decode 的 token 本来就必须算（不算它出不来），没有"可跳过的计算"。
  • 而且 decode 写满的 block 内容 = prompt 前缀 + 自己生成的 token，另一个序列除非采样撞出完全相同的生成 token，否则命中率≈0。

所以 decode 的角色是生产者/注册者：它把自己写满的 block 的 hash 注册进 cache，供未来请求复用——省的是未来的显存，不是自己的计算。

图3 里 decode 写满 block 后算 hash、save it in cached_blocks 就是"注册"。流程里的 Is hash_str in cached_blocks or free_tables? 主要服务于 CoW 分叉出的多序列之间偶然撞 hash 时的去重，不是常规复用路径。

4. KV block 的生命周期管理：驱逐器（Evictor）

驱逐器横跨 prefill 和 decode，不属于某一阶段——它是设备上所有"引用归零但保留待复用"物理块的全局管理者。

核心思想：引用归零的块不立刻销毁，而是进"缓刑区" free_tables 等待被复用。

  • 一个物理块没有任何逻辑块引用时（如某 seq 整个推理做完），本该释放。
  • 但 prefix caching 的优化思想是：留着它，万一不久后来个新 seq 的前缀（如同一段 system message）和它一致，就能直接复用，省下存储和计算。
  • 于是设置驱逐器（evictor），其 free_tables 存放这些暂时不用的物理块。

设备可用物理块的等式（点睛）：

设备全部可用物理块 = 正在被使用/等待使用的块 + free_tables 中的块

💡 free_tables 不是"额外开销"，是延迟释放的复用池。池里的块是"薛定谔的块"——在被复用前一直是驱逐候选：hash 匹配的新请求先到 → 被复用救回；缺空间的建块请求先到 → 被驱逐。free_tables 不是垃圾桶，是缓刑区。

驱逐器的两件职责（它是被动的池子管理者，不决定谁要建块）：

  1. 入池 + 供查：接收引用归零的块，供 prefill 查询复用。
  2. 执行驱逐：别人要建块而设备无空闲时，从 free_tables 踢一个腾位置。

建块时的三道关卡（驱逐只是最后一道）：

prefill / decode 想建一个物理块
  ① 算 hash 查 free_tables：有同 hash 可复用？ → 命中即复用，结束
  ② 没命中 → 设备还有空闲块？ → 有则直接建新块，结束
  ③ 没空闲 → 这才轮到驱逐器：从 free_tables 按策略踢一个，腾空间建新块

驱逐策略（步骤③）：

  1. LRU：优先驱逐最久未使用的块。
  2. max_num_tokens：若多个块 LRU 时间相同，驱逐其 hash 计算涵盖 token 最多的那个（即图例中 num_tokens 最大的——前缀越长越具体、越难被再次命中，优先踢）。
  3. 随机：若 LRU 和 max_num_tokens 仍并列，随机踢一个。

Prefix Caching 易错点速查 List

# 易错点 正确理解
1 hash 算的是 KV？ 不是。hash 算的是 token，所以 prefill 前（KV 未算）就能算出来并查表。
2 hash 只算本 block 的 token？ 不是。是累积前缀 hash（从头到本块所有 token），因为 KV 依赖全部前缀。
3 命中可以跳着复用（跳过中间 miss）？ 不能。必须从头连续命中到第一个 miss 为止，后面 KV 依赖前面内容。
4 prefix caching 只省显存不省 FLOPs？ 错。prefill 命中省计算（TTFT↓）；decode 不省 FLOPs（无可跳过的计算）。别把 decode 结论推广到 prefill。
5 prefill 是"先全算完再发现能复用就丢弃"？ 错。是查表前置、命中跳过计算。先全算完就没省到 FLOPs。
6 decode 阶段是去复用别人（消费者）？ 不是。decode 是生产者/注册者：写满 block 才算 hash 注册进 cache，供未来复用，自己命中率≈0。
7 decode 能像 prefill 一样前置算 hash？ 不能。decode 的 token 是生成出来才知道的，只能写满 block 后算 block-level hash。
8 要不要用 prefix cache 取决于"同步/异步"？ 不是。取决于"前缀 block 是否已被前人算完写满注册进 cache"，与请求何时进 batch 无直接因果。
9 驱逐器只管 prefill / 只管 decode？ 都不是。它横跨两阶段：decode 产出的满块引用归零后入 free_tables，prefill 建块时查复用/触发驱逐。
10 引用归零的块立刻销毁？ 不是。进 free_tables 缓刑区等待复用；只有 prefill 缺空间时才按 LRU→max_num_tokens→随机 驱逐。
11 驱逐器决定"要不要建块"？ 不是。建不建块是调用方（prefill/decode 分配逻辑）的事；驱逐器只在"建块没空间"时被动执行驱逐。

三、Continuous Batching

一句话定位：让请求能动态进出 batch（不再绑死整个请求跑完），并让 prefill 和 decode 共处一个 batch。

强依赖 PagedAttention：请求随时进出意味着显存要随时分配/释放，只有 paged 的 block 抽象能做到"动态进出不产生 external frag"。没有 PagedAttention，continuous batching 一组重排 batch 就会把显存切成碎片。

1. 问题：static batching 的两个病

问题1：decode长短不一
早结束的请求 → 有的请求做完了decode 有的请求还在做decode → 已经完成request的请求 已经做完decode的请求空转 → GPU 气泡。

请求A：要生成 200 个 token
请求B：要生成 5 个 token
绑死在一个 batch → B 第 5 步就结束了，但坑被占着
              → B 的槽位空跑到第 200 步，期间全在算 padding（纯浪费）

问题2：prefill需要右padding
static batching 按"规则矩形张量"组织计算，长度不齐的请求被迫 padding 到最长，算力浪费在废 token 上。prefill 阶段尤其明显，decode阶段因为每个请求都是一个token一个token去算，所以没有浪费。

2. 问题1方案：iteration-level scheduling

一句话定位：request-level改成iteration-level

request-level（static）：组一次 batch → 绑死跑到所有请求结束 → 才换下一批
iteration-level（Orca）：每跑 1 步，prefill+decode都做完了 → 重组 batch（踢走结束的、放进新来的）→ 再跑 1 步

当 t5 时刻发现 seq3 已完成请求，立刻释放其显存槽位，新请求 seq5 得以在同一轮就补入 batch，不再等到整批结束才腾位。

但是！！存在prefill长尾问题，prefill太长拖慢整个iteration速度，所以需要后面的chunked prefill

3. 问题2方案：selective batching

一句话定位：非attention拍平，attention展开

3.1 Attention 层：按请求拆开各算各

💡 拆开 attention 一次性化解了两个"看似不同"的差异：
①prefill不读KV/decode读KV
②各请求decode的index/mask不同

Continuous Batching 易错点速查 List

# 易错点 正确理解
1 static batching 慢只是因为不能动态进出？ 是两个病：①request-level 绑死 → 早结束空占坑（GPU 气泡）；②凑规则张量靠 padding → 算废 token。
2 selective batching 的动机是为了 PP？ 不是。它是 iteration-level scheduling 的自然结果——动态重组后 batch 里天然混着 prefill+decode，必须处理。PP 只是顺便受益的场景。
3 selective batching 是什么高深机制？ 就是该叠的叠、该拆的拆：Linear/LayerNorm 拍平 batch，attention 按请求拆开。“selective” 指的就是这种算子级的区别对待。
4 attention 不能 batch 是因为各请求长度不齐？ 错（流行误解）。根因是 attention 有序列内 token 交互（A 不能 attend B），长度不齐只是表象——padding 也解决不了，因为 pad 位不能让别的请求 attend。Linear 逐 token 独立，长度不齐根本无所谓。
5 Linear 层要 padding 对齐成规则张量？ 不用。变长 token 一行一行叠成 [total_token, H]，H 都一样，直接叠、无 padding。这正是治病二的关键。
6 prefill/decode 在 attention 的三个差异（读KV/长度/mask）要专门解决？ 拆开各算各的就一次性全化解了——这三条只在"塞进同一个 attention 运算"时才是问题，拆开后前提不成立。
7 「prefill并行 vs decode逐token」是 continuous batching 要解决的问题？ 不是。在吞吐视角下它被 GEMM 无害吸收（不疼）；只有切到延迟视角（decode 的 TBT）才显形——那是 chunked prefill 的事。
8 continuous batching 不依赖 PagedAttention？ 依赖。请求动态进出要随时分配/释放显存，靠 paged block 才能"动态进出不产生 external frag"。
9 PD 分离更好，所以 continuous batching / chunked prefill 没必要？ 不对，且这是关键认知。PD 分离动机是 prefill(compute-bound)/decode(memory-bound) 最优配置打架，但代价是跨实例 KV 传输（跨机尤贵，Mooncake 主攻此）。是否分离看 KV 传输代价 vs 干扰+次优配置代价。且两者不同层级不互斥：即便 PD 分离，prefill 池/decode 池内部仍在用 continuous batching。chunked prefill（时间交错）与 PD 分离（空间隔离）是同一个病的两种药方。

四、Chunked Prefill

一句话定位：把一个长 prefill 切成若干小块，分多个 iteration 喂。

1. 核心目的：解决 prefill 长尾问题，拖垮整个batch的iteration

continuous batching 让 prefill 和 decode 混进同一个 iteration。但调度只在 iteration 边界发生：一个长 prefill（如 2000 token）会让这一个 iteration 算得巨久，这步没结束之前，同 batch 的 decode 请求出不了下一个 token → 用户端看到 TBT（time-between-tokens）飙高、卡顿。

做法：把长 prefill 切成贴着统一 token budget（如 512）的小块，每块单独和 decode 混进一个 iteration。于是每个 iteration 都又短又均匀，decode 步步稳定吐 token。

2. 附带品：PP 的 bubble

这是副产品，不是设计目的。 因为 chunked prefill 把每个 micro-batch 的 token 数削成统一上限，间接磨平了 PP 里「micro-batch 耗时不齐」导致的 bubble。

2.1 PP 的三个 bubble 是什么

  • PB1：prefill micro-batch 之间序列长度不一（A 的 prompt 长、B 短）。
  • PB2：prefill 重 vs decode 轻（prefill 一个 micro-batch 几百 token，decode 每请求只 1 token）。
  • PB3：decode micro-batch 之间规模不一（X 有 16 请求、Y 只 4 请求；各自 KV 长度也不同）。

2.2 chunked prefill 怎么解决其中的两个 bubble

唯一武器：给每个 micro-batch 设统一 token budget，把耗时磨平。 判据——「这个 PB 的不齐，能不能被『削 prefill 的 token 数』磨平」：

  • PB1：长的 prefill 被切成等大的块，长短差异被磨平，stage 间对齐。
  • PB2：① prefill 被切到和 decode 同量级；② 每个 micro-batch 内「prefill 块 + decode」混合凑到同一 budget，不再有「纯重 prefill」对「纯轻 decode」的对立。
  • PB3 → 暂时治不了
