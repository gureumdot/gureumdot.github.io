RLHF 四件套：PPO 完整流程、GRPO 去掉 Critic 的简化版、DPO 核心 loss、LoRA 低秩适配。

LoRA（Low-Rank Adaptation）

旁路矩阵 A·B（rank=r），B 必须全零初始化，推理时 merge 进主干无延迟。

import torch
import torch.nn as nn

class LoRALinear(nn.Module):
    def __init__(self, in_dim, out_dim, r=8, alpha=16):
        super().__init__()
        
        self.linear = nn.Linear(in_dim, out_dim, bias=False)
        self.linear.weight.requires_grad = False  # 冻结主干权重
        
        self.A = nn.Parameter(torch.randn(in_dim, r))
        self.B = nn.Parameter(torch.zeros(r, out_dim))  # B 必须全零初始化
        
        self.scale = alpha / r

    def forward(self, x):
        lora_out = x @ self.A @ self.B
        return self.linear(x) + self.scale * lora_out

    def merge(self):
        # 推理融合：消除旁路延迟
        # nn.Linear.weight 是 (out_dim, in_dim)，注意转置
        self.linear.weight.data += (self.A @ self.B).T * self.scale

PPO（RLHF-PPO）

import torch
import torch.nn as nn

class PPOTrainer:
    def __init__(self, actor, critic, ref, reward_model,
                 actor_optimizer, critic_optimizer,
                 kl_ctl=0.1, gamma=1.0, lam=0.95,
                 clip_range=0.2, clip_range_value=0.2,
                 clip_reward_value=5.0, ppo_epochs=4):
        self.actor = actor                        # 策略网络（被训练）
        self.critic = critic                      # 价值网络（被训练）
        self.ref = ref                            # 参考模型（冻结，算 KL）
        self.reward_model = reward_model          # 奖励模型（冻结，打分）
        self.actor_optimizer = actor_optimizer
        self.critic_optimizer = critic_optimizer
        self.kl_ctl = kl_ctl
        self.gamma, self.lam = gamma, lam
        self.clip_range = clip_range
        self.clip_range_value = clip_range_value
        self.clip_reward_value = clip_reward_value
        self.ppo_epochs = ppo_epochs              # 同一批数据复用多少轮

    # ---------- 1. 拼 token 级即时奖励：KL 惩罚 + 末位 RM 分数 ----------
    def compute_rewards(self, prompts, log_probs, ref_log_probs, reward_score, action_mask):
        """R = KL_Penalty(逐 token) + Final_Reward(只加在最后一个有效 token)"""
        # KL 惩罚：防止 Actor 偏离 Ref 太远
        kl_divergence_estimate = -self.kl_ctl * (log_probs - ref_log_probs)
        rewards = kl_divergence_estimate

        # 定位 response 的起止位置
        start = prompts.shape[1] - 1
        ends = start + action_mask[:, start:].sum(dim=1) + 1

        # RM 分数截断，避免极端值
        reward_clip = torch.clamp(reward_score, -self.clip_reward_value, self.clip_reward_value)

        # 把最终分数加到 response 最后一个有效 token 上
        batch_size = log_probs.shape[0]
        for j in range(batch_size):
            rewards[j, start:ends[j]][-1] += reward_clip[j]
        return rewards

    # ---------- 2. GAE：从后往前 DP，同时得到 advantage 和 return ----------
    def get_advantages_and_returns(self, values, rewards, start):
        lastgaelam = 0
        advantages_reversed = []
        length = rewards.size()[-1]
        for t in reversed(range(start, length)):
            nextvalues = values[:, t + 1] if t < length - 1 else 0.0
            # TD Error: delta = r_t + gamma * V_{t+1} - V_t
            delta = rewards[:, t] + self.gamma * nextvalues - values[:, t]
            # A_t = delta_t + gamma * lambda * A_{t+1}
            lastgaelam = delta + self.gamma * self.lam * lastgaelam
            advantages_reversed.append(lastgaelam)
        advantages = torch.stack(advantages_reversed[::-1], dim=1)
        returns = advantages + values[:, start:]   # return = advantage + value
        return advantages.detach(), returns

    # ---------- 3. Actor / Critic loss ----------
    def actor_loss_fn(self, logprobs, old_logprobs, advantages, mask):
        log_ratio = (logprobs - old_logprobs) * mask
        ratio = torch.exp(log_ratio)
        pg_loss1 = -advantages * ratio
        pg_loss2 = -advantages * torch.clamp(ratio, 1.0 - self.clip_range, 1.0 + self.clip_range)
        return torch.sum(torch.max(pg_loss1, pg_loss2) * mask) / mask.sum()

    def critic_loss_fn(self, values, old_values, returns, mask):
        values_clipped = torch.clamp(values,
                                     old_values - self.clip_range_value,
                                     old_values + self.clip_range_value)
        vf_loss1 = (values - returns) ** 2
        vf_loss2 = (values_clipped - returns) ** 2
        return torch.sum(torch.max(vf_loss1, vf_loss2) * mask) / mask.sum()

    # ---------- 4. train_step：一批 rollout 数据，复用 ppo_epochs 轮更新 ----------
    def train_step(self, batch):
        """batch 里都是 rollout 阶段冻结下来的旧量：old_logprobs / old_values / advantages / returns"""
        seq, mask = batch["sequences"], batch["action_mask"]
        old_logprobs = batch["old_logprobs"]      # rollout 时 actor 算的，固定不变
        old_values   = batch["old_values"]        # rollout 时 critic 算的，固定不变
        advantages   = batch["advantages"]        # GAE 算好的，已 detach
        returns      = batch["returns"]

        # 关键：同一批数据复用多轮。old_* 始终不变，只有 actor/critic 在每轮被更新
        for _ in range(self.ppo_epochs):
            # 用当前（最新）actor/critic 重新前向，得到 new_logprobs / new_values
            new_logprobs = self.actor(seq).log_prob(seq)
            new_values   = self.critic(seq)

            actor_loss = self.actor_loss_fn(new_logprobs, old_logprobs, advantages, mask)
            self.actor_optimizer.zero_grad()
            actor_loss.backward()
            self.actor_optimizer.step()

            critic_loss = self.critic_loss_fn(new_values, old_values, returns, mask)
            self.critic_optimizer.zero_grad()
            critic_loss.backward()
            self.critic_optimizer.step()

易错点

前提1：PPO 的哲学是"不论方向是正确还是错误，都不能更新过度，也不能更新太慢"

前提2：更新参数靠梯度。clip 之后 loss 变成常数 → 梯度为 0 → 自动刹车。所以"想刹车"就意味着"选 clip 那条 loss"，因为它能让梯度归零。

易错点1：为什么 advantages 要 .detach()，但 returns 不用？

因果链（按这个顺序记）：

  1. 梯度流到谁，谁被更新。
  2. 我们要：policy loss 只更新 Actor，value loss 只更新 Critic。
  3. advantages 身上挂着 Critic 的计算图（它由 values 推导而来），却要进 policy loss → 不处理就会让 policy loss 的梯度反流进 Critic（错）→ 所以 .detach() 把 Critic 这条路砍掉。Actor 经由 ratio 照常被训练。
  4. returns 这里因为advantages 被切除→ returns只跟values有关 → 不用 detach。

易错点2（Actor）：为什么 pg_loss = max(loss1, loss2)，而不是 min？

嵌套易错点：为什么 clip 之后梯度就是 0？

用链式法则展开：∂loss/∂w = (∂loss/∂y_clip) · (∂y_clip/∂y) · (∂y/∂w)，关键在中间这一段。

∂y_clip/∂y 就是 clamp 函数的斜率：

  • y 在 [low, high] 区间内（没被卡）：y_clip = y，这段 = 1 → 链路通，梯度照常 2(y-y')·x。
  • y 超出区间（被卡死）：y_clip = constant（常数），这段 = 0 → 整条链乘以 0 → ∂loss/∂w = 0。

所以 clip 不是"软化"梯度，而是在边界外直接切断反向传播。选了 clip 那条 loss，ratio 一旦越界，梯度立刻归零，更新完全停止。

actor更新策略总结：

更新目标更新过头更新过慢
adv > 0 & 提高概率 & 提高 logprobs_new超过 1 + ε & 需要刹车 & 选择clip低于 1 - ε & 需要加速 & 选择非clip
adv < 0 & 降低概率 & 降低 logprobs_new低于 1 - ε & 需要刹车 & 选择clip高于 1 + ε & 需要加速 & 选择非clip

易错点2（Critic）：为什么 vf_loss = max(loss1, loss2)，也用的是max，逻辑和 Actor 一样吗？

完全对称，只是方向维度变了，actor模型的更新方向由优势决定，而critic模型的更新方向由真实值与预测值的差值决定

更新目标更新过头更新过慢
values < returns & 提高 values超过 old_values + ε & 需要刹车 & 选择clip低于 old_values - ε & 需要加速 & 选择非clip
values > returns & 降低 values低于 old_values - ε & 需要刹车 & 选择clip高于 old_values + ε & 需要加速 & 选择非clip

易错点2：为什么 Actor loss 用 ratio = new/old 的 surrogate 形式，而 Critic loss 直接用 (values - returns)²，不涉及 old_logprobs？

根本原因是两者扮演的角色不同：

  1. Actor 需要模拟与环境交互：所以loss里面需要由旧策略的设计，使得新策略能够模仿旧策略来和环境交互，并且还能限制新策略不要跑太远
  2. Critic 只做回归预测：critic模型不需要模仿，只需要不断的更新预测即可；而且critic这里已经有真实值来限制预测值不要跑太远了，所以不需要引入旧预测来限制新预测

DPO（Direct Preference Optimization）

最简洁的对齐算法，无需 Reward Model 和 Critic，直接对 preference pair 优化。

import torch
import torch.nn.functional as F

def dpo_loss(policy_chosen_logp, policy_rejected_logp, 
             ref_chosen_logp, ref_rejected_logp, beta=0.1):
    # 当前模型 vs 参考模型的对数概率差
    chosen_diff = policy_chosen_logp - ref_chosen_logp
    rejected_diff = policy_rejected_logp - ref_rejected_logp
    
    # 好回答的相对优势 - 坏回答的相对优势
    logits = chosen_diff - rejected_diff
    
    loss = -F.logsigmoid(beta * logits).mean()
    return loss

GRPO（Group Relative Policy Optimization）

GRPO 真正动的就两处——advantage 来源、KL 位置

手撕

import torch

class GRPOTrainer:
    def __init__(self, actor, ref, actor_optimizer,
                 G=8, clip_range=0.2, kl_ctl=0.04, grpo_epochs=1):
        self.actor = actor                  # 策略网络（被训练）
        self.ref = ref                      # 参考模型（冻结，算 KL）
        self.actor_optimizer = actor_optimizer
        self.G = G                          # 每个 prompt 采样多少条
        self.clip_range = clip_range
        self.kl_ctl = kl_ctl
        self.grpo_epochs = grpo_epochs       # 同一批数据复用几轮

    # ---------- 1. 组内 advantage：reward 进来，组内 z-score ----------
    def compute_advantages(self, rewards):
        """rewards: (B,G)"""
        mean = rewards.mean(dim=1, keepdim=True)
        std = rewards.std(dim=1, keepdim=True)
        advantages = (rewards - mean) / (std + 1e-8)    # 组内 z-score
        return advantages.detach()         # (B, G) 每条回答一个标量

    # ---------- 2. loss：PPO 式 clip + KL 独立正则 ----------
    def compute_loss(self, new_logprobs, old_logprobs, ref_logprobs,
                     advantages, action_mask):
        # advantage 是 sequence-level 标量，广播到每个 token
        advantages = advantages.unsqueeze(-1)                  # (B, G) → (B, G, 1) → 广播 (B, G, R)

        # PPO 式 clipped surrogate（这部分和 PPO 的 actor loss 完全一样）
        log_ratio = (new_logprobs - old_logprobs) * action_mask
        ratio = torch.exp(log_ratio)
        pg_loss1 = - advantages * ratio
        pg_loss2 = - advantages * torch.clamp(ratio, 1 - self.clip_range, 1 + self.clip_range)
        pg_loss = torch.max(pg_loss1, pg_loss2)        # (B*G, R)

        # KL 正则：直接加进 loss（不进 reward），逐 token
        kl = ref_logprobs - new_logprobs                # (B*G, R)

        loss = (pg_loss + self.kl_ctl * kl) * action_mask
        return loss.sum() / action_mask.sum()

    # ---------- 3. train_step：一批 rollout 数据，复用多轮更新 ----------
    def train_step(self, batch):
        """batch 里 old_logprobs / ref_logprobs / rewards 都是 rollout 时冻结的"""
        seq, mask = batch["sequences"], batch["action_mask"]
        old_logprobs = batch["old_logprobs"]            # rollout 时 actor 算的，固定
        ref_logprobs = batch["ref_logprobs"]            # ref 算的，固定
        rewards = batch["rewards"]                      # 规则打分，(B, G)

        # advantage 只跟 reward 有关，整批算一次即可（不像 ratio 每轮变）
        advantages = self.compute_advantages(rewards)

        for _ in range(self.grpo_epochs):
            new_logprobs = self.actor(seq).log_prob(seq)   # 当前 actor 重新前向
            loss = self.compute_loss(new_logprobs, old_logprobs, ref_logprobs,
                                     advantages, mask)
            self.actor_optimizer.zero_grad()
            loss.backward()
            self.actor_optimizer.step()

易错点

易错点1：advantage 是 sequence-level 标量，不是 token-level —— 这是没 Critic 的直接后果

PPO 有 Critic 给每个 token 估 value，GAE 才能算出 token 级 advantage。GRPO 砍掉 Critic 后，一条回答只有一个整体得分，组内归一化出来的也是一条一个标量，只能广播到该回答的所有 token（同一条里每个 token 共享同一个 advantage）。
→ 粒度从「逐 token」退化成「逐回答」，这是 GRPO 用简洁换掉的精度。

易错点2：KL 为什么能从 reward 里挪出来当独立 loss 项？关键看「有没有 GAE 这条时序传播链」

PPO 的 KL 必须逐 token 揉进 reward，因为它要喂给 GAE 做时序信用分配（KL 惩罚也得跟着 advantage 沿序列回传）。
GRPO 没有 GAE、没有时序回传，advantage 就是个组内标量 → KL 留在 reward 里没有任何传播意义 → 直接当 loss 正则项更干净。
一句话：KL 放哪，取决于下游有没有 GAE。

易错点3：KL 项的符号别搞反

KL 是惩罚项：actor 越偏离 ref，loss 应该越大。代码里 kl = ref_logprobs - new_logprobs，当 actor 拉高某 token 概率（new_logprobs ↑）偏离 ref 时，kl ↓……
→ 所以真正要记的是：它和 pg_loss 同号相加，整体最小化时会把 actor 往 ref 拽。手撕时先确认「偏离 ref → loss 增大」这个方向对，符号自然不会错。

易错点4：train_step 里 advantage 在循环外算，ratio 在循环内算

advantages 只依赖 rewards（冻结量），整批算一次就固定了，放循环外。
ratio = new/old 里的 new_logprobs 每轮都被更新，必须每轮重新前向算，放循环内。
→ 别把 advantage 误放进循环（白算）、也别把 new_logprobs 误提到循环外（ratio 永远等于 1，等于没训练）。

易错点5：KL 用的是 new 策略，不是 old —— 它和 PPO 的 KL 含义已经不一样了

表面看 GRPO 只是把 KL 从 reward 挪到 loss，其实连「算 KL 用哪个策略」都变了：

PPO 的 KL GRPO 的 KL
位置 reward 里 loss 里（独立项）
用谁算 old_logprobs（rollout 冻结） new_logprobs（当前策略，每轮重算）
多轮更新 不变（揉进 reward 就冻死了） 每轮都变

根因是 KL 的身份变了：

  • PPO：KL 是 reward 的一部分（惩罚信号）。reward 是采样那一刻的客观事实，本就该冻结，所以用 old。
  • GRPO：KL 是 loss 的正则项（约束「当前」策略别偏离 ref）。约束的对象是不断被更新的 new 策略 → 必须每轮拿最新的 new 重算，否则就是在约束一个已经不存在的旧策略。

成本几乎为零：new_logprobs 本来每轮就要为 ratio 重算，KL 只是多一个 ref - new 的减法。

易错点6：GRPO 的根本劣势 —— 只能 sequence-level，知道「整条有问题」但不知道「哪一步有问题」

一句话：采样求平均必须「同一状态重复出现」才能平均，但中间状态条条发散、互不重复，唯一被所有回答共享的状态只有起点 prompt → 所以平均优势只能在「整条生成完、回到共享起点」这个层面算一次，凑不出单步那个组。

对比着记：

Critic：对任意单个状态都能直接吐 V(s)，哪怕这状态只出现一次（靠网络泛化预测）
        → 能做单步（token-level）优势
GRPO ：靠「一组共享同一状态的回答求平均」估基线
        → 只有起点 prompt 被所有回答共享 → 只能整条算一次（sequence-level）
        → 中间状态各走各的、不重复 → 凑不出组 → 做不了单步优势

后果：一条「中间走错、结尾纠正回来」的回答，最终 reward 是对的，整条共享一个正 advantage → 那个走错的 token 也被连带鼓励。GRPO 区分不出功劳和过错具体在哪一步。
→ 这正是 long-CoT 场景下 GRPO 信用分配粗糙的根源，也是后续 PRM（过程奖励）、token-level 变体想补的洞。
