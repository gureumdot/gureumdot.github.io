一、问题与 naive 的两个版本

转置 B[j][i] = A[i][j]。逻辑一行的事，难点全在访存：输入按行连续、输出按列连续，行列在转置里天然对立，naive 无论如何都有一头跳跃。

naive 有两个写法，区别就在"把跳跃放在读端还是写端"：

// NaiveRow：按行读、按列写
__global__ void transposeNaiveRow(float* A, float* B, const int M, const int N) {
  int ix = blockIdx.x * blockDim.x + threadIdx.x;
  int iy = blockIdx.y * blockDim.y + threadIdx.y;
  if (iy < M && ix < N) { B[ix * M + iy] = A[iy * N + ix]; }
}

// NaiveCol：按列读、按行写
__global__ void transposeNaiveCol(float* A, float* B, const int M, const int N) {
  int ix = blockIdx.x * blockDim.x + threadIdx.x;
  int iy = blockIdx.y * blockDim.y + threadIdx.y;
  if (iy < N && ix < M) { B[iy * M + ix] = A[ix * N + iy]; }
}

用 warp 视角拆（warp 内 x 递增 → ix 递增、iy 不变）：

读 A 写 B
NaiveRow B[ix*M+iy]=A[iy*N+ix] ix 在低位 → 连续 ✓ ix 在高位（×M）→ 相差 M，跳跃 ✗
NaiveCol B[iy*M+ix]=A[ix*N+iy] ix 在高位（×N）→ 跳跃 ✗ ix 在低位 → 连续 ✓

一句话：Row = 读连续写跳跃，Col = 写连续读跳跃。 两个都逃不掉"必有一头跳跃"——这正是下一步要用 shared memory 解决的：让读写两头都连续。

⚠️ 面试爱抠：Row 和 Col 谁快？ 一般 NaiveRow（跳跃写）更快。因为 GPU 的写有 write-back 合并缓冲，跳跃写的延迟更容易被掩盖；而跳跃读会直接拖慢后续依赖它的计算（读延迟在关键路径上）。但两者都远不如 shared memory 版，这个差异属于"锦上添花"级别的细节。

二、最终优化版本

直接给选定版本（模板版，含全部三个优化），先看代码和调用，再逐条拆。

template<int Bm, int Bn>
__global__ void transposeSharedSwizzling(float* A, float* B, const int M, const int N) {
  __shared__ float tile[Bm][Bn];

  /* -------- 读取阶段 -------- */
  int r0 = blockIdx.y * Bm;
  int c0 = blockIdx.x * Bn;
#pragma unroll
  for (int y = threadIdx.y; y < Bm; y += blockDim.y) {
    int r = r0 + y;
    if (r >= M) break;
#pragma unroll
    for (int x = threadIdx.x; x < Bn; x += blockDim.x) {
      int c = c0 + x;
      if (c < N) { tile[y][x ^ y] = A[r * N + c]; }
    }
  }

  __syncthreads();

  /* -------- 写入阶段 -------- */
#pragma unroll
  for (int y = threadIdx.y; y < Bn; y += blockDim.y) {
    int c = c0 + y;
    if (c >= N) break;
#pragma unroll
    for (int x = threadIdx.x; x < Bm; x += blockDim.x) {
      int r = r0 + x;
      if (r < M) { B[c * M + r] = tile[x][x ^ y]; }
    }
  }
}

调用配置：

constexpr int Bm = 32, Bn = 32;
dim3 block(32, 8);                          // x=32 对齐 warp，y=8 → 每线程搬 32/8=4 个元素
dim3 grid((N + Bn - 1) / Bn, (M + Bm - 1) / Bm);
transposeSharedSwizzling<Bm, Bn><<<grid, block>>>(A, B, M, N);
//                       ├模板参数┤ ├─launch配置─┤ ├─函数实参─┤

💡 block 形状本身就是优化：blockDim.x = 32 保证一个 warp 恰好覆盖 32 列、coalescing 完美对齐；blockDim.y = 8 让 block 有 8 个 warp，每线程用循环（y += blockDim.y）搬 4 个元素。若 blockDim.x < 32，一个 warp 会横跨多行、x 不再独占连续维，coalescing 立刻打折。

优化点逐条拆解

优化 1：一个 tid 负责多个元素（打满算力）

代码里的 for (y = threadIdx.y; y < Bm; y += blockDim.y) 双层循环就是它——block 是 32×8，tile 是 32×32，所以 y 方向要循环 32/8 = 4 次，一个线程负责 4 个元素。

目的：用更少的线程/block 覆盖更大的 tile，提高每线程的计算访存比，减少 block 调度开销，把算力喂饱。

优化 2：shared memory → 读写两头都连续

这是 transpose 的核心。先把术语钉死（极易接反）：

  • HBM = GPU 显存（A/B，慢、大、远）。global memory 在此。coalescing 是 HBM 的概念——按"连续地址凑一次事务"工作。
  • SRAM = 片上高速存储（shared memory / L1L2 / 寄存器）。shared memory 是 SRAM。bank 是 shared memory 的概念。

💡 为什么 shared memory 能解决问题：coalescing 只对 HBM 有要求。shared memory 没有 coalescing 这回事——它按 bank 工作，一次访问只要 32 个地址落在 32 个不同 bank，就是一次事务，完全不要求地址连续。于是把"HBM 必须连续"这个硬约束，转移到 shared memory 上：HBM 两头（读 A、写 B）都连续，跳跃被吸进 shared memory，而 shared memory 用 bank 机制天然消化掉了跨行访问。

转置发生在哪？ 写进 shared 时按 [行][列] 正常存，从 shared 读出来时按转置读（读一列当一行用），再正常按行写回 HBM。即"列读行写"。

为什么写 HBM 必须按行：HBM 没有 bank、只认地址连不连续，所以写出去那头必须连续 → 只能"列变行"。

读写两阶段为什么 x/y 绑的轴会反？（最绕、最值得讲透）

行列语义是程序员自由定义的，coalescing 不强制语义。但只有"让 threadIdx.x 绑到当前要连续那个矩阵的低位"才能 coalesce：

阶段 要哪个 HBM 连续 该矩阵低位（连续维） x 绑 y 绑
读 A A A 的列 列 c=c0+x 行 r=r0+y
写 B B B 的行（转置后行才是低位） 行 r=r0+x 列 c=c0+y

所以 x 在读阶段绑列、写阶段绑行——不是 bug，是为 coalescing 主动牺牲了"两阶段语义一致"。你也可以写"语义工整版"（两阶段都 r=r0+y,c=c0+x），结果完全正确，但写 B 那头会变成跳跃写、慢约 32 倍。

而 shared 的索引：shared 第一维永远是 shared 的行（= A 的行 = B 的列），第二维永远是 shared 的列。

  • 写 shared：数据来自 A，A 的行是 y → tile[y][...]
  • 读 shared：要写到 B[c][r]，而 B[c][r]=A[r][c]，在 shared 里坐标 =（A 行, A 列）=（r=r0+x, c=c0+y）→ 第一维是 x → tile[x][...]

这就是 tile[y][..] 写、tile[x][..] 读、第一维从 y 变 x 的来源——第一维的 y→x 切换本身就是转置。

优化 3：swizzle（x ^ y）解决 bank conflict

引入 shared memory 后冒出第三个问题：bank conflict。若 tile 是 [32][32]、按列访问，同一列的 32 个元素地址相差 32，全部落到同一个 bank → 32 路冲突、串行 32 次。

两条解决路线：

  • padding（tile[32][33]，+1）：加一列，把列元素物理错开到不同 bank。代价：浪费显存 + tile 不再是 2 的幂。
  • swizzle（tile[y][x ^ y]）：不加列，用 列 ^ 行 打乱列在 bank 里的落点，让同一逻辑列散到不同 bank。零显存浪费，tile 保持 [32][32]。

x ^ y 的妙处是对称：按行扫（x 变 y 固定）不撞 bank，按列扫（y 变 x 固定）也不撞 bank——所以读写两阶段能共用同一个 x ^ y 表达式。

核心：更换每一列的存储位置，进而达到按虽然列读取数据，但由于存储位置不同而导致bank不同，所以可以合并为一次内存事务，解决bank conflict

三、易错点速查 List

# 易错点 正确理解
1 threadIdx 的 x/y 对应矩阵行列？ x/y 是笛卡尔坐标（x 横 y 纵）。换算矩阵：y=行，x=列。
2 warp 内"x 连续变化"是无条件的？ 不是。warp 按线性 ID（x 最低位、行主序）每 32 个切一刀。只有 blockDim.x ≥ 32 时一个 warp 才落在"同 y、x 连续 0~31"；否则横跨多行、y 也变。
3 "按行/按列读取"是时间顺序？ 不是。SIMT 无先后。它描述同一拍 32 个线程同时访问的地址在空间上连成横/竖，是空间形状不是时间次序。
4 NaiveRow 和 NaiveCol 哪个是 naive？ 都是。Row=读连续写跳跃，Col=写连续读跳跃，都有一头跳跃。
5 naive 能靠调 block 形状救回来？ 不能。转置行列对立，读连续则写必跳，根因是布局不是配置。要 shared memory 中转。
6 HBM 就是 shared memory / SRAM？ 接反了。HBM=显存（慢大远，coalescing 概念）；SRAM=片上（shared/cache/寄存器，bank 概念）。shared memory 是 SRAM。
7 shared memory 也要考虑 coalescing？ 不要。shared memory 没有 coalescing，只有 bank。跨行访问只要不撞 bank 就是一次事务，不要求连续。
8 转置发生在写 shared 还是读 shared？ 写 shared 正常存，读 shared 时转置（列读），再按行写回 HBM。即"列读行写"。
9 读写两阶段 x/y 绑的轴一样？ 不一样。x 始终绑"当前要连续矩阵的低位"——读阶段绑 A 的列、写阶段绑 B 的行。为 coalescing 主动牺牲语义一致性。
10 行列语义是 warp/硬件强制的？ 不是。语义程序员自由定，结果都对。但只有 x 绑低位的那种定法能 coalesce，其它定法慢约 32×。
11 tile[y][..] 写、tile[x][..] 读写错了？ 没错。shared 第一维=shared 行（=A 行=B 列）。写来自 A 用 y，读去往 B 用 x，第一维 y→x 的切换就是转置。
12 swizzle 和 padding 是一回事？ 解决同一问题（bank conflict）的两条路线。padding 物理错开（费显存）；swizzle 用 x^y 打乱落点（零浪费、tile 保持 2 的幂）。
13 swizzle 读写要用不同表达式？ 不用。x^y 对称，按行扫、按列扫都不撞 bank，两阶段共用同一 x^y。
