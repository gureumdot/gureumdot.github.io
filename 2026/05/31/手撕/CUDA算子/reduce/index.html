Reduce

推理优化 CUDA 算子优化
手撕 CUDA算子
Publish Date:   2026-05-31

手撕目标

Reduce = 把一个数组归约成一个标量（sum / max / mean）。LLM 的 fused kernel 里全是它：

  • RMSNorm / LayerNorm：每个 block 处理一个 token 的 hidden 向量（如 4096 维），先 reduce 出 Σx²。
  • Softmax（LM head，vocab≈15 万）：reduce 出行 max 和 Σexp。
  • loss / 范数 / 相似度：本质都是 reduce。

所以 reduce kernel 的这套优化 = 你写任何 norm / softmax fused kernel 的地基。

我们要手撕的版本是融合了 grid-stride + 两层 WarpReduce 的 reduce8 最终版，它把 warp shuffle、shared memory 极简化、模板展开全部融合在一起，shared 只存 NUM_WARPS 个值、全程只 sync 一次。

手撕代码（reduce8 最终版，逐行批注）

下面是 reduce 的最终手撕版本。优化层级在后面逐节展开，这里先整体过一遍代码。

#define WARP_SIZE 32
#define CEIL(a, b) (((a) + (b) - 1) / (b))

// ★ reduce4 warp shuffle | reduce5 模板展开
// kWarpSize 是编译期常量，#pragma unroll 把循环完全展开 → 无运行时循环/分支
// shfl_down = 漏斗，结果只在 lane 0；若需每个 lane 都拿结果（RMSNorm/Softmax）改用 shfl_xor
template <int kWarpSize = WARP_SIZE>
__device__ __forceinline__ float warpReduce(float val) {
#pragma unroll
    for (int delta = kWarpSize >> 1; delta >= 1; delta >>= 1)
        val += __shfl_down_sync(0xffffffff, val, delta);
    return val;
}

// ★ reduce5 模板展开：blockSize 是编译期常量 → NUM_WARPS 编译期确定 → warpReduce<NUM_WARPS> 完全展开
template <int blockSize>
__global__ void reduce8(float* d_A, const int N) {
    // ★ reduce8 两层 WarpReduce：shared 只存 NUM_WARPS 个部分和，非整棵归约树
    extern __shared__ float data[];
    int tid = threadIdx.x;
    int i   = blockSize * blockIdx.x + tid;

    // ★ reduce6 grid-stride：grid 大小与 N 解耦，一个线程吃多个元素，寄存器累加
    float sum = 0.f;
    for (int index = i; index < N; index += blockSize * gridDim.x)
        sum += d_A[index];

    // ★ reduce4 warp shuffle | reduce8 两层 WarpReduce：每个 warp 各自先归约，纯 shuffle 无 sync
    sum = warpReduce<WARP_SIZE>(sum);

    // ★ reduce8 两层 WarpReduce：lane 0 把 warp 部分和写进 shared，只写 NUM_WARPS 个值
    if ((tid & (WARP_SIZE - 1)) == 0) data[tid / WARP_SIZE] = sum;

    // ★ reduce2 顺序寻址（思想继承）：全程唯一一次 __syncthreads，shared 归约彻底退场
    __syncthreads();

    // ★ reduce8 两层 WarpReduce：warp 0 收尾；超出 NUM_WARPS 的 lane 补 0，满足 mask 要求
    constexpr int NUM_WARPS = CEIL(blockSize, WARP_SIZE);
    if (tid < 32) {
        sum = tid < NUM_WARPS ? data[tid] : 0.f;
        // ★ reduce4 warp shuffle | reduce5 模板展开：第二层 WarpReduce，NUM_WARPS 是编译期常量
        sum = warpReduce<NUM_WARPS>(sum);
    }

    // ★ reduce3 load 先加（思想继承）：tid==0 写出 block 部分和，供第二轮 kernel 收尾
    if (tid == 0) d_A[blockIdx.x] = sum;
}

手撕要点速查：

  • warpReduce<NUM_WARPS> 第二层的 mask 0xffffffff 要求 warp 0 的 32 个 lane 全部活跃，所以超出 NUM_WARPS 的 lane 必须补 0，否则读到未定义值。
  • 不要忘记写同步！整个 block 只有一次 __syncthreads，所有归约主体在寄存器 shuffle 里完成。

调用配置（两轮发射，见第七章 reduce6）：

constexpr int blockSize = 256;
int grid = 216;                                            // 按 occupancy 定、与 N 解耦
size_t smem = CEIL(blockSize, WARP_SIZE) * sizeof(float);  // shared 只存 NUM_WARPS 个部分和
reduce8<blockSize><<<grid, blockSize, smem>>>(d_A, N);     // 第一轮：N → grid 个部分和
//     ├模板参数┤ ├───────launch配置───────┤ ├函数实参┤
reduce8<blockSize><<<1, blockSize, smem>>>(d_A, grid);    // 第二轮：grid → 1，一个 block 收尾

〇、总览：优化阶梯

优化阶梯一句话总览（后面逐级展开，每级配图 + 改前改后代码）：

版本 核心动作 解决的问题 最终版用到
reduce0 朴素交错寻址 （起点）warp divergence
reduce1 活跃线程压到前排 消 divergence，但引入 bank conflict
reduce2 顺序寻址（stride 从大到小） 消 bank conflict，但首轮半数线程空闲
reduce3 load 时先加一次 消首轮空闲，block 数减半
reduce4 最后一个 warp 用 warp shuffle 去 __syncthreads + 去 shared memory 中转 ✓
reduce5 模板完全展开 去循环开销 + 运行时分支（把 s 变编译期常量） ✓
reduce6 grid-stride loop grid 与 N 解耦：发射次数稳定为 2、可打满 occupancy ✓
reduce7 float4 访存 减指令条数（叠加在 grid-stride 上，非替代）
reduce8 写 shared 前先做一轮 WarpReduce 最终版：shared 只存 warp 数个值 ✓

reduce8 直接包含了 reduce4（warp shuffle）、reduce5（模板展开）、reduce6（grid-stride），三者都在最终版里；reduce3 的 load 先加思路已被 grid-stride 的寄存器累加彻底替代（更泛化的写法），单独标 ✓ 反而会误导；reduce7（float4）是可叠加但未写入的一招。

一、reduce0：朴素交错寻址 → warp divergence

for (int s = 1; s < blockDim.x; s <<= 1) {
    if ((tid % (s * 2)) == 0) { data[tid] += data[tid + s]; }
    __syncthreads();
}

活跃线程 = 0, 2s, 4s, …，在一个 warp 内活跃 lane 和闲置 lane 交错排列。

⚠️ 问题不是"有线程闲着"（归约本来就会越来越多线程闲）。问题是活跃 / 闲置 lane 在同一个 warp 里交错 → warp 必须把 if 的两个分支串行各跑一遍（SIMT 一次只能走一条路径）→ warp divergence。早期迭代最严重（半数 lane 在每个 warp 里穿插）。

二、reduce1：活跃线程压到前排 → 消 divergence，但 bank conflict

for (int s = 1; s < blockDim.x; s <<= 1) {
    int index = 2 * s * tid;                       // 改：不再用 tid 直接寻址
    if (index < blockDim.x) { data[index] += data[index + s]; }
    __syncthreads();
}

活跃线程变成 tid = 0,1,2,… 连续一段 → 整个 warp 要么全活跃要么全闲，分支不再在 warp 内分裂 → divergence 消失。

⚠️ 代价：index = 2s·tid 让线程以 stride 2s 访问 shared memory。s=1 时 thread 0/1/2… 落在 bank 0/2/4…，thread 0 与 thread 16 撞同一个 bank → 2-way bank conflict；s 越大冲突越重。换句话说：divergence 治好了，bank conflict 冒出来了。

💡 bank conflict 速记：shared memory 分 32 个 bank，warp 内 32 线程同拍访问，若多个线程命中同一 bank 的不同地址，硬件只能串行服务 → 慢。连续访问（线程 i 访问地址 i）天然无冲突。

三、reduce2：顺序寻址 → 消 bank conflict

for (int s = blockDim.x / 2; s > 0; s >>= 1) {     // 改：stride 从大到小
    if (tid < s) { data[tid] += data[tid + s]; }
    __syncthreads();
}

stride 从 blockDim/2 开始逐步折半。活跃线程 tid < s 连续，且每个线程访问 tid 与 tid+s——相邻线程命中相邻 bank → 无 bank conflict，同时无 divergence。

⚠️ 代价：首轮只有前一半线程（tid < blockDim/2）干活，另一半从一开始就闲着。这是下面 reduce3 要榨干的浪费。

四、reduce3：load 时顺手加一次 → 消首轮空闲 + block 数减半

reduce2 首轮半数线程闲——那就让每个线程一次读两个元素、读的时候就加起来，一个 block 因此覆盖 2·blockDim 个元素，load 阶段无人空闲。

int i = blockIdx.x * blockDim.x * 2 + tid;          // 一个 block 管 2*blockDim 个元素
data[tid] = (i < N              ? d_A[i]              : 0.f)
          + (i + blockDim.x < N ? d_A[i + blockDim.x] : 0.f);   // load 顺手加一次
__syncthreads();
// 之后接 reduce2 的顺序寻址归约

💡 全局看：每个 block 干两倍的活 → block 总数减半 → 整个 grid 少一半"读全局 + 多轮 sync 归约"的开销。这是 reduce6 grid-stride（一个 thread 扛 n 个元素）的雏形。

五、reduce4：最后一个 warp 用 warp shuffle → 去 sync + 去 shared

这是从"shared memory 归约"跨到"warp 原语归约"的关键一跃。

核心观察：当 s ≤ 32，所有活跃线程都落在 warp 0 内部。warp 是 SIMT 锁步执行的——这几轮根本不需要 __syncthreads，也不必走 shared memory。直接用 warp shuffle 让 lane 之间寄存器对寄存器传值。

5.1 四个 shuffle 函数（对着图）

__shfl_*_sync(mask, var, arg, width)：每个 lane 拿自己的 var，按规则去读别的 lane 的 var，结果落在自己的寄存器里。

__shfl_sync(_, x, srcLane, w)——所有 lane 读同一个 srcLane（广播）：

width 把 warp 切成若干独立小组，各组各自广播：

__shfl_up_sync(_, x, d, w)——lane i 读 i−d（向高位搬，前 d 个 lane 保留自己），用于 prefix scan：

__shfl_down_sync(_, x, d, w)——lane i 读 i+d（向低位搬），reduce 用它：

__shfl_xor_sync(_, x, m, w)——lane i 与 i^m 互换（蝶形），all-reduce 用它：

5.2 down vs xor 做 WarpReduce

down = 漏斗（结果只在 lane 0）：

val += __shfl_down_sync(_, val, 16/8/4/2/1)，每轮把后半折叠加到前半。5 轮（满 warp）后只有 lane 0 拿到完整和，其余 lane 是中间值。reduce 只要一个总和、由 tid==0 写出，所以用 down 足够。

xor = 蝶形（每个 lane 都有结果）：

val += __shfl_xor_sync(_, val, 16/8/4/2/1)，每轮和异或伙伴互换并相加。5 轮后每个 lane 都拿到完整和（这就是 all-reduce）。

💡 怎么选（用真 kernel 记）：

  • 纯 sum reduce → down：结果只写一次。
  • RMSNorm / softmax → 必须 all-reduce（xor）：算出 Σx² / Σexp 后，每个线程都要拿它去归一化自己负责的那几维，不是只有 lane 0 用 → 和必须回到所有 lane。

一句话：结果只写一次 → down；每个线程都要拿结果继续算 → xor。

（图里画的是 8-lane 示例所以只有 4→2→1 三步；满 warp 是 16→8→4→2→1 五步。）

5.3 代码：改前 → 改后

// 改前（reduce2/3）：最后 6 轮还在 shared memory 里跑，每轮都 sync
for (int s = blockDim.x/2; s > 0; s >>= 1) {
    if (tid < s) data[tid] += data[tid + s];
    __syncthreads();
}

// 改后（reduce4）：归约到剩 64 个就交给 warp shuffle，无 sync、无 shared（设 blockDim ≥ 64）
for (int s = blockDim.x/2; s > 32; s >>= 1) {
    if (tid < s) data[tid] += data[tid + s];
    __syncthreads();
}
float sum = (tid < 32) ? data[tid] + data[tid + 32] : 0.f;   // 64→32 折进寄存器
if (tid < 32) sum = warpReduce<32>(sum);                      // 32→1 全 shuffle，无 sync
if (tid == 0) d_A[blockIdx.x] = sum;
// WarpReduce（down 版，#pragma unroll + 模板 → 编译期完全展开）
template <int kWarpSize = WARP_SIZE>
__device__ __forceinline__ float warpReduce(float val) {
#pragma unroll
    for (int mask = kWarpSize >> 1; mask >= 1; mask >>= 1)
        val += __shfl_down_sync(0xffffffff, val, mask);
    return val;
}

⚠️ mask 不是摆设（面试常踩）：0xffffffff 表示"这 32 个 lane 都参与"。shuffle 要求被读的 lane 此刻必须活跃且在 mask 里，否则读到未定义值。这正是 reduce8 里 sum = tid < NUM_WARPS ? data[tid] : 0.f 要补 0 的原因——warp 0 做最后归约时，超过 NUM_WARPS 的 lane 在 shared 里没有有效数据，得先用 0 垫掉再 reduce。

六、reduce5：模板完全展开

把 blockSize 做成模板参数，循环边界变成编译期常量 → 编译器把 for 循环和 if (blockSize >= N) 全部展开成无分支直线代码，去掉循环计数、条件判断的运行时开销。

template<int blockSize>
__global__ void reduce5(...) {
    // ... load ...
    if (blockSize >= 512) { if (tid < 256) data[tid] += data[tid+256]; __syncthreads(); }
    if (blockSize >= 256) { if (tid < 128) data[tid] += data[tid+128]; __syncthreads(); }
    if (blockSize >= 128) { if (tid <  64) data[tid] += data[tid+ 64]; __syncthreads(); }
    float sum = (tid < 32) ? data[tid] + data[tid + 32] : 0.f;
    if (tid < 32) sum = warpReduce<32>(sum);
    if (tid == 0) d_A[blockIdx.x] = sum;
}

6.1 一堆 if 怎么就等价于 for 了？——破最大误解

第一眼看这段最容易误解成"只 reduce 了一次，剩下的怎么办"。关键在于 if (blockSize >= 512 && tid < 256) 里有两个性质完全不同的判断：

if ( blockSize >= 512  &&  tid < 256 )
       └─ 编译期常量          └─ 运行期变量
       blockSize 是模板参数，   每个线程跑到这才判断，
       编译时就知道真假        决定这轮哪些线程干活
  • blockSize >= N：编译期就被算成 true/false。false 的话整行连同它的 __syncthreads 被编译器物理删除，GPU 上根本不存在这条指令。
  • tid < N：运行期判断，决定这一轮哪些线程干活。

最大的误解是盯着 tid < 256 以为"只有前 256 个线程干活、剩下 if 执行不了"。错——剩下的 if 不是给别的线程的，是给同一批线程的下一轮 reduce 的。 这些 if 是顺序执行的层层递进，不是互斥分支。

以 blockSize = 512 为例，编译器逐行判断 blockSize >= N 后，真实保留的代码是：

if (tid < 256) data[tid] += data[tid+256];  __syncthreads();   // 512→256
if (tid < 128) data[tid] += data[tid+128];  __syncthreads();   // 256→128
if (tid <  64) data[tid] += data[tid+ 64];  __syncthreads();   // 128→64
if (tid <  32) data[tid] += data[tid+ 32];  __syncthreads();   // 64→32
// 然后 warpReduce 处理 32→1

这正是你以为"缺失"的层层递进：每行就是一轮 reduce，stride 256→128→64→32 逐步折半，和 reduce2 的 for 循环做的事一模一样，只是手动摊开成固定几行。第一行 256 个线程活完，数据已压到前 256 个位置，第二行 tid<128 去加 data[tid+128] 因此合法。blockSize >= 1024 这行因为 512>=1024=false 被整行删掉了。

模板源码（两种判断混在一起）

if (blockSize>=1024 && tid<512) …
if (blockSize>=512 && tid<256) …
if (blockSize>=256 && tid<128) …
if (blockSize>=128 && tid<64 ) …
if (blockSize>=64 && tid<32 ) …
紫=编译期常量(blockSize≥N)    蓝=运行期(tid<N)

编译期

编译后真实保留的代码（blockSize=512）

≥1024 行：512≥1024=false → 整行删除
if(tid<256) d[tid]+=d[tid+256]; sync; // 512→256
if(tid<128) d[tid]+=d[tid+128]; sync; // 256→128
if(tid<64 ) d[tid]+=d[tid+64 ]; sync; // 128→64
if(tid<32 ) d[tid]+=d[tid+32 ]; sync; // 64→32

为什么"剩下的 if 执行得了"？ 误解：盯着 tid<256 以为"只有前 256 个线程干活，后面 if 执行不了"。 真相：后面的 if 不是给别的线程的，是给同一批线程的下一轮 reduce的——它们是顺序执行的层层递进，不是互斥分支。 第一行 256 个线程加完，数据已压到前 256 个位置；第二行 tid<128 去加 d[tid+128] 因此合法。stride 256→128→64→32 逐步折半。 本质：和 reduce2 的 for 做的事一模一样，只是把循环手动摊开成固定几行 → 省掉循环管理指令。

6.2 为什么展开适合编译器优化？——优化前 vs 优化后

核心：展开本身不"优化"，它只是把循环变量 s 从运行期变量变成编译期常量，从而解锁编译器一连串激进优化。

原来的 for (s = blockDim/2; s > 32; s >>= 1) 里 s 是运行期变量，编译器不敢假设它的取值，只能保守地一轮一轮老实跑。展开成 if 后 stride 全是写死的字面常量（256/128/64/32），编译器能做这几件 for 时做不了的事：

维度 for（s 是变量，编译器保守） 展开后（stride 是常量，编译器放手）
循环管理 每轮跑 s>32 比较 / s>>=1 移位 / 跳转，全保留 无 s → 循环管理指令全部消失
地址计算 地址带变量 s → 先算 tid+s 再访存 偏移是常量 → 编进 load 指令的立即数字段
无用分支 blockSize 未知 → 所有分支保留、运行时判断 恒假分支（如 >=1024）连同 __syncthreads 整个删除
指令调度 动态循环挡住重排，寄存器/调度难规划 指令序列编译期定死 → 可重排 + 隐藏访存延迟

💡 真正省的是：循环控制指令 + 变量地址计算 + 保留无用分支——这些在 GPU 上每线程每轮叠加起来才显著。

6.3 展开不是免费午餐：代价 + 何时该展开

⚠️ 很多人只记"展开=快"，不知道它有膨胀上限。

展开会让指令数膨胀（code size 变大），可能挤爆 **instruction cache（I-cache）**反而变慢。所以是否展开要看：

  • reduce 这种：循环只有 5~6 层、每层访存是大头 → 展开收益 >> 膨胀代价，划算，可全展开。
  • 几百次迭代、循环体很大的循环：全展开会把 I-cache 撑爆，得不偿失 → 这时选**部分展开（#pragma unroll 4/8）**而非全展开。

💡 这也回答了"完全展开在不在最终版"：在。但它不再是教科书里"展开 shared memory 归约树"——因为最后一个 warp 已经被 warp shuffle 接管，warpReduce<> 模板本身就是全展开的；reduce5 展开的只剩 blockDim/2 → 64 这几层 shared 归约。

七、reduce6：grid-stride loop（网格跨步循环）

一句话：让一个线程负责多个元素，跨步遍历整个数组，先在寄存器里把这些元素累加好，再进 shared 归约。

float sum = 0.f;
for (int i = blockIdx.x*blockDim.x + tid; i < N; i += gridDim.x*blockDim.x)
    sum += d_A[i];        // 寄存器累加，步长 = grid 总线程数
// 之后 sum 进 shared / warp 归约

为什么步长是 gridDim.x*blockDim.x（grid 总线程数）？因为这样同拍里相邻线程仍访问相邻地址 → 全局访存保持 coalesced；下一拍整个 grid 一起往后跳一整段。

7.1 核心红利：grid_size 与 N 彻底解耦

reduce6 之前（reduce0~4），每个 block 固定吃 2×blockSize 个元素，所以 block 数被 N 绑死：

grid_size = ceil(N / (2×block_size))   ← N 一亿、block=256 → 被迫开 ~19.5 万个 block

reduce6 之后，跨步循环让一个线程能吃任意多个元素，于是 grid_size 不再由 N 决定，你可以自己定一个固定的、合适的值，循环兜底吃下任意长的 N：

不管 N 多大，i += 总线程数 一直跑到 i >= N 为止
N 一亿就多跑几圈，N 一千就一圈结束 → grid 大小完全不用动

⚠️ 措辞要准：不是"block 数随便、无所谓"，而是"block 数从被 N 逼出来，变成由你主动按 GPU 选最优"。解耦的价值正是让你能挑一个最优 grid_size，而不是被 N 逼出一个烂的。

7.2 grid_size 怎么选 + 为什么不能太大/太小

最优 grid_size 不是按 N 算，而是按 **GPU 占用率（occupancy）**算：

grid_size ≈ SM 数量 × 每个 SM 能驻留的 block 数
（如 A100 有 108 个 SM、每 SM 驻留 2 个 block → 开 216 个 block 刚好填满）
  • 太大（如 reduce3 被迫开 19.5 万）：第一轮吐出 19.5 万个部分和，收尾还要 reduce 一大堆中间结果。
  • 太小（如只开 10 个）：① SM 空着、带宽吃不满（reduce 是 memory-bound，直接慢）；② 单线程要跑的圈数暴涨（1e8/(10×256)≈4 万圈），延迟掩盖不住。
  • 甜点：大到刚好填满 GPU（occupancy 饱和），又不至于留下海量部分和。实践用 cudaOccupancyMaxActiveBlocksPerMultiprocessor 让 runtime 算，而非手拍。

7.3 隐藏功劳：把"多次 kernel 发射"压成"两次"

发射次数取决于第一轮输出多少部分和，而这 = 第一轮的 block 数：

第一轮输出的部分和数 = block 数
  ≤ 1024（一个 block 能收完）→ 第二轮一个 block 搞定 → 共 2 次发射
  > 1024（一个 block 收不完）→ 还要再分多轮 → 多次发射
  • reduce6 之前：block 数被 N 绑死成 19.5 万 → 第二轮一个 block 收不完 → 763 → 3 → 1，多次 kernel launch（跨 block 没有同步原语，只能靠重启 kernel 让上一轮结果全局可见）。
  • reduce6 之后：block 数按 occupancy 开成比如 216 → 216<1024 → 第二轮一个 block 一口气收完 → 稳定两次发射。

💡 跨 block 为什么必须重启 kernel：__syncthreads() 只能同步一个 block 内的线程，block 之间在同一次 launch 内没有任何同步原语。要让所有 block 的部分和对下一轮可见，只能结束当前 kernel、再启动一次（kernel 之间有隐式全局同步）。

★ 跨 block 为什么要重启 kernel：__syncthreads 只同步 block 内；block 间同一次 launch 无同步原语，只能结束 kernel 再启动（kernel 间隐式全局同步）。
★ reduce6 的价值主要不是"快"（见性能表它和 reduce4 同速），而是工程灵活性 + 能打满 occupancy。

7.4 三个好处总结

  1. grid 大小与 N 解耦：grid 按 GPU 占用率开，不按 N 开 → 部分和少、发射次数稳定为 2。
  2. 拿廉价的寄存器加法换昂贵的 shared 归约层级：一个线程把 N/总线程数 个元素先在寄存器里加掉，省去多层 shared 归约树（算法层面的 cascading）。
  3. 访存连续对齐，带宽利用率高。

💡 工程价值 vs 提速：reduce6 的价值主要不在单次"快多少"（见后面性能表，它和 reduce4 几乎同速），而在工程鲁棒性——同一份 kernel 不为不同 N 重编/重开 grid，且能主动打满 occupancy。它是灵活性优化，不是单点提速。

八、reduce7：向量化访存（vectorized load）

让一个线程一条指令读 4 个连续 float（float4），而不是 4 条指令读 4 次：

float4 v = reinterpret_cast<const float4*>(d_A)[idx];
sum += v.x + v.y + v.z + v.w;

8.1 它到底影响什么：减"指令条数"，不减"字节数"

要读 4 个 float，无论如何都得搬 16 字节——字节数固定，变的是用几条指令搬完：

标量版：  LDG.32 [addr+0] / +4 / +8 / +12   ← 4 条独立 load，各自发射、各算地址
float4：  LDG.128 [addr]                    ← 1 条指令搬 128 bit = 16 字节

影响两件事：

  1. 指令数 ÷4 → 缓解指令发射瓶颈：GPU 每周期能发射的指令有限，数据量大时 load 指令会塞满调度器；一条 LDG.128 顶四条，发射压力降到 1/4。
  2. 单条指令搬更多字节 → 更易打满带宽（MLP↑）：同时在飞的内存请求越大，越容易喂满总线带宽。

💡 不改变 coalescing：32 个线程各读自己的 float4，只要地址连续照样合并——一个 warp 一拍合并出 32×16=512 字节的大事务。

8.2 global 能用，shared 也能用——但收益不同

❓ 向量化只能用在读显存（global）、不能用在 SRAM（shared）？原理上 shared 也能 float4，不是不能用。

但两者收益不同，所以实践中 float4 主要往 global 上招呼：

  • global：延迟几百周期、带宽是 reduce 的瓶颈，指令数和事务大小直接决定快慢 → 收益大。
  • shared：本身就快（几十周期），reduce 在 shared 阶段的瓶颈是 bank conflict 和 sync，不是指令发射；而且 float4 一次横跨连续 4 个 bank，反而可能加重 bank conflict。所以 shared 阶段一般不刻意 float4。

一句话：不是"shared 不能用"，而是"shared 用了收益小、还可能踩 bank conflict，不划算"。

8.3 关键：float4 与 grid-stride 正交，应叠加而非二选一

很多教学代码（包括常见的 reduce6_vec4）为了单独演示 float4，把 grid-stride 的跨步循环砍掉，写成"每线程固定读一个 float4、一个 block 固定吃 4×blockSize"——这等于把 grid_size 又绑回 N 了（grid=ceil(N/(4×bs))），上一节辛苦的解耦丢了。

正确认知：两者解决正交的问题，应当合体——

grid-stride：管"一个线程吃几个元素"（循环结构）→ grid 与 N 解耦
float4：     管"一条指令搬几个元素"（访存宽度）→ 指令数 ÷4
// 正确：跨步循环的壳 + 每步 float4 的核
float sum = 0.f;
for (int i = blockIdx.x*blockDim.x + tid; i*4 < N; i += gridDim.x*blockDim.x) {
    float4 r = FLOAT4(d_A[i*4]);
    sum += r.x + r.y + r.z + r.w;          // 一步吃 4 个，循环跨步吃任意多
}
// 尾部不足 4 个标量兜底 ...

⚠️ 别把 reduce 的优化理解成"一版替换一版"的线性升级——float4 是叠加在 grid-stride 之上的正交优化，不是替代它。

8.4 边界 + 对齐（面试给代码挑错爱埋）

if (i < N - 4) { float4 r = FLOAT4(d_A[i]); ... }   // ⚠️ 边界写窄了

FLOAT4(d_A[i]) 读 i, i+1, i+2, i+3，最大下标 i+3 要求 i+3 <= N-1，即 i <= N-4。写成 i < N-4（等价 i <= N-5）会把 i==N-4 这个本来合法的整块错误踢进标量分支——不致命但少用一次向量化。严格应是 if (i <= N-4) 或 if (i+3 < N)。

⚠️ 另一个硬前提：FLOAT4 要求起始地址 16 字节对齐。i = 4×(...) 保证 i 是 4 的倍数 → 地址对齐 ✓。地址不对齐会非法访问或掉性能。

💡 收益前提：float4 是锦上添花。reduce 在 reduce4 已经把带宽打到 ~97%（见性能表），向量化在单次大数组 reduce 里几乎看不出提速——它的价值在某些访存未打满、或需要压指令发射的场景。

9.1 为什么 reduce4 之后就不动了：撞内存带宽墙

reduce 是 memory-bound，耗时几乎完全由"搬多少数据 + 带宽多快"决定。reduce4 已把带宽打到 96.92%，离物理天花板（100%）只剩 3%——瓶颈从"代码写得好不好"变成"显存物理带宽就这么宽"。reduce5/6/7 优化的是指令开销、循环结构、sync 次数，可它们在总耗时里本就只占百分之几，再省也榨不出明显时间。

⚠️ reduce4 之后那些 251.x（251.68 / 251.52 / 251.39 / 251.74）的差异是测量噪声（0.3us / 0.1%），跑两次顺序都能反过来。别读成"reduce4 最优、reduce8 更差"。

9.2 那 reduce5/6/7 白做了吗？——不，是这张表的测试条件没给它们舞台

这是最重要的认知（面试追问点）：这张表用的是一个对 reduce4 最友好的固定配置（block=256、单次 reduce、N 固定），在这个点上 reduce4 已撞带宽墙。换条件，后面的优化才显价值：

  • reduce5（模板展开）/ reduce8（warp 实现 block reduce）：单次调用里省下的指令/sync 开销被几百 us 的访存淹没，占比 0.x%，看不见。但真实 AI 模型里每个 RMSNorm/Softmax 的 reduce 在每一层、每个 step 都跑，几万次累加，这 0.x% 就成了实打实的墙钟时间。memory-bound 单次看不出，高频调用累积才显形。
  • reduce6（grid-stride）：价值根本不在这张表的"快不快"，而在工程灵活性（grid 与 N 解耦、不重编）+ 能打满 occupancy。表里 N 固定，这优势体现不出来——它是鲁棒性优化，不是单点提速。

💡 一句话：不能因为表里数字没涨就说 reduce5/6/7 没用——是这张表的单点测试条件压根没给它们发挥的维度（指令占比、grid 灵活性、多级归约的发射开销）。

十、reduce8：最终版 = 写 shared 前先 WarpReduce

把 “BlockReduce” 拆成两层 WarpReduce，shared memory 退化成只存"每个 warp 一个部分和"。

流程（对应你贴的代码）：

① grid-stride：每个线程把负责的多个元素累加进寄存器 sum
② 每个 warp 各自 WarpReduce → 每个 warp 出 1 个部分和（纯 shuffle，无 sync）
③ 各 warp 的部分和按 warpId 写进 shared（只写 NUM_WARPS 个值）
④ __syncthreads（全程仅此一次）
⑤ warp 0 读回这 NUM_WARPS 个值（不足补 0）→ 再 WarpReduce<NUM_WARPS> → block 总和
⑥ tid==0 写出
template<int blockSize>
__global__ void reduce8(float* d_A, const int N) {
    extern __shared__ float data[];
    int tid = threadIdx.x;
    int i   = blockSize * blockIdx.x + tid;

    // ① grid-stride 累加进寄存器
    float sum = 0.f;
    for (int index = i; index < N; index += blockSize * gridDim.x) sum += d_A[index];

    // ② 每个 warp 先 WarpReduce
    sum = warpReduce<WARP_SIZE>(sum);
    // ③ 按 warpId 写 shared
    if ((tid & (WARP_SIZE - 1)) == 0) data[tid / WARP_SIZE] = sum;

    __syncthreads();    // ④ 仅此一次

    // ⑤ warp 0 归约各 warp 的部分和
    constexpr int NUM_WARPS = CEIL(blockSize, WARP_SIZE);
    if (tid < 32) {
        sum = tid < NUM_WARPS ? data[tid] : 0.f;     // 补 0：见 5.3 的 mask 易错点
        sum = warpReduce<NUM_WARPS>(sum);
    }
    if (tid == 0) d_A[blockIdx.x] = sum;             // ⑥
}

为什么这是终点：shared memory 只承载 NUM_WARPS（如 8）个值、只用一次，彻底没有多层 shared 归约树 → shared 流量和 sync 都压到最低，所有归约都在 warp shuffle（寄存器）里完成。

reduce8 用到的优化清单：grid-stride（①）+ warp shuffle WarpReduce（②⑤）+ 两层 BlockReduce 结构 + 模板完全展开（warpReduce<> / template<blockSize>）。唯独没显式做向量化访存——这是它还能再叠加的一招（把①的标量 load 换成 float4）。

10.1 数据从头到尾不落地：全程在寄存器

reduce8 最反直觉、也最关键的一点：从 grid-stride 到两层 WarpReduce，数据始终待在寄存器里，显存只被读、从不写回，shared 几乎不碰。

逐段拆：

float sum = 0.f;                                   // sum 是每线程私有标量 → 寄存器
for (int index = i; index < N; index += stride)
    sum += d_A[index];                             // 从【显存】读一个元素，加进【寄存器】sum
sum = warpReduce<WARP_SIZE>(sum);                  // 寄存器 ↔ 寄存器 shuffle，不碰 shared
  • sum 是普通局部标量、每线程私有、没取地址、不是数组 → 编译器一定放进寄存器。
  • grid-stride 每圈做的是"读显存一个元素 → 加到寄存器 sum"，全程只有读显存,没有任何写显存。读进来的 d_A[index] 进临时寄存器、加完即丢，sum 自始至终不落地。
  • __shfl_down_sync 的本质是 warp 内 lane↔lane 的寄存器直连搬运——不经过 shared、不经过显存。

所以完整数据路径是：显存(只读) → 寄存器累加 → 寄存器间 shuffle → shared 中转 NUM_WARPS 个值(唯一一次) → 寄存器 shuffle → 出。

💡 这解释了 reduce8 为什么对 bank conflict “免疫”：bank conflict 是 shared 独有的问题。reduce8 的归约主体全在寄存器 + shuffle 通路里完成，几乎不进 shared，所以 reduce0/1/2 处理的那些坑在这条路径上根本不会产生——不是"被解决了"，是"产地被拆了，问题无从发生"。

🔗 为什么不能完全干掉 shared：warp 内 32 lane 的寄存器可以靠 shuffle 硬件直连，但两个不同 warp 之间的寄存器没有任何直接通路（shuffle 的作用域就是一个 warp）。8 个 warp 各出的部分和要汇到 warp 0，唯一的桥就是 shared——所以 shared 删不掉，但能退化成"只存 NUM_WARPS 个值、写一次读一次"。

10.2 唯一碰 shared 的那一步，为什么也不冲突

reduce8 唯一碰 shared 的地方：8 个 warp 各把部分和写进 data[0..7]，sync 一次，warp 0 读回 data[0..7]。为什么这步没有 bank conflict？

理由只有一个词——连续（stride=1）。data[0]~data[7] 这 8 个 float 落在 bank 0~7（地址以 4 字节为单位，对 32 取模 = bank 号），各占一个独立 bank；warp 0 读回时 lane i 读 data[i]，8 个 lane 各打各的 bank，无任何两 lane 撞同一 bank。

⚠️ 这步的理由不是"只有 8 个数据",也不是"warp 和 bank 都是 32"。 bank conflict 的唯一直接判据是 “同一 warp 的 32 lane 同一拍的地址、模 32 后 bank 号是否重复”，由 lane 间 stride 决定：stride=1 永不冲突，偶数 stride 必冲突。数据总量(32 还是 1024)、block 大小,统统不进入判定——硬件每次只发射一个 warp，只看这 32 个 lane 这一拍的地址。

两个反例钉死这点：① 32 个数据、stride=2 写法（data[2*tid]+=data[2*tid+1] 配上够多 lane）→ 照样冲突；② 1024 个数据、stride=1（data[tid]）→ 完全不冲突。所以"标准折半 reduce 不冲突"的真因是它每轮都 stride=1,不是数据少。 reduce1 冲突 = index=2s·tid 让 stride=2s（偶数）；reduce8 这步免冲突 = stride=1。

10.3 reduce8 和前面几版到底什么关系（破"前面优化白做了"）

最容易误判的就是"reduce8 把 block 内归约重构了，前面 reduce0/1/2/5 是不是白学了"。逐个对账：

前面的版本 在 reduce8 里的归宿
reduce0/1/2（divergence / bank conflict / 顺序寻址） 它们处理的问题根源都在那个 for(s){ if(tid<s) shared折半; sync; } 循环。reduce8 把这个循环整个删了，换成寄存器累加 + shuffle → 两个问题在新路径上根本不产生，无需解决。不是优化消失，是问题产地被拆。
reduce5（模板完全展开） 在。block 内 shared 折半的展开没了（那段循环删了），但 warpReduce<> 内部的 mask 折半 for 循环靠 template<int> + #pragma unroll 完全展开。展开思想完整保留，只是展开对象从"shared 归约树"换成"warp shuffle 序列"。
reduce6（grid-stride） 在，就是流程①那行跨步循环。grid 与 N 解耦、稳定两次发射的红利全继承。
reduce4（warp shuffle） 在且升级。reduce4 只在最后一个 warp 调一次 shuffle；reduce8 是每个 warp 都调 + 收尾再调一层（两层 WarpReduce 拼出 BlockReduce）。
reduce7 float4 向量化 reduce8 没显式写——这是唯一可再叠加的一招（与 grid-stride 正交，见第八节）。

💡 一句话：没有任何优化消失。 变的只是 block 内那棵归约树的中间几层——reduce4 是"shared 折半到 32 → 末端一个 warp shuffle 收尾"；reduce8 是"全员 warp 各 shuffle 出部分和 → 一个 warp 收尾"，shared 从"承载整棵树"退化成"只存 NUM_WARPS 个中转值、只碰一次"。主体循环没了，所以代码"看起来全改了"，但解决的事和用的招一个没变。

十、面试要写到哪种程度？

结论：至少写到 warp shuffle 版（reduce4 级别），这是现在的"及格线 / 加分项"分界。

# 易错点 正确理解
1 reduce0 的问题是"有线程闲着"？ 不是。归约本就越来越多线程闲。问题是活跃 / 闲置 lane 在 warp 内交错 → 分支串行 → warp divergence。
2 reduce1 把活跃线程压前，代价是什么？ divergence 消失，但 index=2s·tid 造成 stride 访问 → bank conflict。治好一个冒出另一个。
3 reduce2 顺序寻址就完美了？ 无 conflict 无 divergence，但首轮半数线程空闲（reduce3 来榨干）。
4 reduce4 为什么最后一个 warp 能去掉 __syncthreads？ 因为 s≤32 时活跃线程全在一个 warp 内，warp 锁步执行天然同步，且 shuffle 走寄存器、不碰 shared。
5 down 和 xor 的 WarpReduce 一样？ 不一样。down=漏斗，结果只在 lane 0；xor=蝶形，每个 lane 都有结果（all-reduce）。
6 reduce 该用 down 还是 xor？ 纯 sum 用 down（写一次）；RMSNorm / softmax 用 xor（每个线程都要拿和归一化自己那几维）。
7 shuffle 的 mask（0xffffffff）可有可无？ 不可。被读 lane 必须活跃且在 mask 内，否则未定义值。这就是 reduce8 要补 0 的原因。
8 "完全展开"在最终版里没有了？ 有。warpReduce<> 模板和 template<blockSize> 就是编译期全展开；只是展开对象从 shared 归约树变成 warp shuffle 序列。
9 reduce5 一堆 if 是"只 reduce 一次"？ 不是。blockSize>=N 是编译期常量（删恒假分支），tid<N 是运行期（选活跃线程）。保留的多行 if 就是层层递进的多轮 reduce，等价于 for。
10 展开为什么快？是展开本身在优化？ 不是展开本身。展开把循环变量 s 从运行期变量变成编译期常量，才解锁：删循环管理指令 + 地址编进立即数 + 删恒假分支 + 可重排。
11 循环就该无脑全展开？ 不。展开膨胀 code size、可能撑爆 I-cache。reduce 层数少划算；几百次大循环体该用部分展开（#pragma unroll 4/8）。
12 grid-stride 的步长为什么是 grid 总线程数？ 这样同拍相邻线程仍访问相邻地址 → 保持 coalesced；整体每拍跳一整段。它让 grid 大小与 N 解耦，用寄存器加法换 shared 归约层级。
13 grid_size 解耦后随便设个固定值就行？ 不。太小则 occupancy 不足 + 单线程圈数暴涨；太大则部分和爆炸。甜点≈SM 数×每 SM 驻留 block 数，用 cudaOccupancyMaxActiveBlocksPerMultiprocessor 算。
14 reduce6 之前是不是要算"刚好够 reduce 的 block 数"？ 是。之前 block 数被 N 绑死（ceil(N/(2×bs))），N 大→block 爆→部分和收不完→多次 kernel 发射。reduce6 后 block 按 occupancy 开→稳定两次发射。
15 跨 block 归约为什么要重启 kernel？ __syncthreads 只同步一个 block 内，block 之间同一次 launch 内无同步原语。要让所有 block 部分和全局可见，只能结束 kernel 再启动（kernel 间隐式全局同步）。
16 向量化访存减少的是字节数？ 不是。字节数固定（4 个 float=16B）。减的是指令条数（÷4）→ 缓解发射瓶颈 + 更易打满带宽。不改变 coalescing。
17 向量化只能用在 global、不能用 shared？ 原理上 shared 也能。但 shared 收益小（本就快、瓶颈是 conflict/sync），且 float4 横跨 4 个 bank 可能加重 conflict，故不划算。
18 float4 是 grid-stride 的替代/升级？ 不是。两者正交：grid-stride 管循环结构（grid 与 N 解耦），float4 管访存宽度（指令数）。应叠加——跨步循环内每步读 float4。常见 vec4 示例砍掉 grid-stride 是教学简化，会把 grid 绑回 N。
19 float4 边界 i < N-4 对吗？ 窄了。读 i~i+3 要求 i<=N-4，写 i<N-4 会漏掉 i==N-4。且 FLOAT4 要求地址 16B 对齐（i=4×... 已保证）。
20 reduce4 之后数字不动 = reduce5/6/7 没用？ 错。reduce4 已撞带宽墙（97%），单次看不出差别；reduce5/7 在高频调用累积才显形，reduce6 价值在工程灵活性 + occupancy。是测试条件没给它们舞台。
21 reduce4 比 reduce8 快（251.68 vs 251.74）？ 不是。0.1% 差异是测量噪声，跑两次能反过来。撞带宽墙后这些数字不可比。
22 reduce8 的 shared memory 还存整个 block 的数据？ 不。只存 NUM_WARPS 个 warp 部分和、只用一次，没有多层 shared 归约树。
23 手撕 reduce 写到 reduce2 就够？ 不够。主推默写 reduce8（grid-stride + 两层 warpReduce，最完整）；白板翻车的退路是 reduce2 骨架 + 末段 warpReduce（=reduce4，及格线）。
24 reduce8 重构了 block 内归约，前面 reduce0/1/2/5 白学了？ 没白学。reduce0/1/2 的问题（divergence/bank conflict）根源在那个 shared 折半循环，reduce8 把循环删了→问题不产生而非"被解决"；reduce5 的展开还在（warpReduce<> 的 mask 折半靠 template+#pragma unroll 全展开，只是展开对象变了）。
25 reduce8 grid-stride 时数据每加一次要写回显存？ 不。sum 是寄存器，全程"读显存→加进寄存器"，只读不写。到 warpReduce 也是寄存器间 shuffle，数据从头到尾不落地，这才是它绕开 bank conflict 的根。
26 bank conflict 由"数据总量"决定（32 不冲突、1024 冲突）？ 错。判定单位永远是一个 warp 的 32 lane，与 block 大小、数据总量无关。唯一判据：这 32 lane 同一拍的地址模 32 后 bank 号是否重复，根源是 lane 间 stride≠1。反例：32 个数据 stride=2 照样冲突；1024 个数据 stride=1 完全不冲突。
27 "做 reduce"这个目标本身就能免 bank conflict？ 不能。reduce2 免冲突是因为它特意用 stride=1 的连续寻址；reduce1 同样在做 reduce，用了 2s·tid 偶数 stride 就冲突。是寻址写法救的，不是"做 reduce"救的。
28 divergence 的代价是"占资源让别的 warp 排队等"？ 不是。GPU 靠几十个 warp 驻留 + scheduler 零开销穿插切换（latency hiding），不会让别的 warp 干等。代价是每个 warp 自己的指令量翻倍（串行跑两条分支）→ 全体总吞吐需求上升 → 整体变慢。伤害"乘在每个 warp 头上"，非"一个拖累一群"。（例外：__syncthreads 屏障处快 warp 会等慢 warp，但那是 barrier 传染，仍非"占 slot"。）
29 reduce2"半数线程空闲"会拉低 occupancy？ 不会。occupancy 是资源占用的静态指标，block 上 SM 后 32 个 warp slot 占满整个生命周期，与"某轮干没干活"无关。闲置的整 warp 只是被发射后走一条掩码全 0 的空指令，几乎免费——这跟 reduce0 的 warp 内交错 divergence（串行跑两遍）有本质区别。
30 warpReduce 都在一个 warp 了，为何不一步把 31 个 lane 全加到 lane 0？ 做不到 + 不划算。shuffle 是"每 lane 读一个源 lane"，没有"多 lane 同时写一个目标"的原语；逐个累加要 31 步且会读到半成品，折半树只需 log₂32=5 步且每步源 lane 都已是完整部分和。
