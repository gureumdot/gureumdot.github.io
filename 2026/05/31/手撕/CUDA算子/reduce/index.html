推理优化 CUDA 算子优化
手撕 CUDA算子
Publish Date:   2026-05-31

〇、总览：reduce 是什么 + 为什么 LLM 里到处是它

reduce = 把一个数组沿某一维归约成一个标量（sum / max / mean）。LLM 的 fused kernel 里全是它：

  • RMSNorm / LayerNorm：每个 block 处理一个 token 的 hidden 向量（如 4096 维），先 reduce 出 Σx²（或均值、方差）。
  • Softmax（LM head，vocab≈15 万）：reduce 出行 max 和 Σexp。
  • loss / 范数 / 相似度：本质都是 reduce。

所以 reduce kernel 的这套优化 = 你写任何 norm / softmax fused kernel 的地基。

优化阶梯一句话总览（后面逐级展开，每级配图 + 改前改后代码）：

版本 核心动作 解决的问题
reduce0 朴素交错寻址 （起点）warp divergence
reduce1 活跃线程压到前排 消 divergence，但引入 bank conflict
reduce2 顺序寻址（stride 从大到小） 消 bank conflict，但首轮半数线程空闲
reduce3 load 时先加一次 消首轮空闲，block 数减半
reduce4 最后一个 warp 用 warp shuffle 去 __syncthreads + 去 shared memory 中转
reduce5 模板完全展开 去循环开销 + 运行时分支
reduce6 grid-stride loop 一个 thread 扛多个元素，grid 大小与 N 解耦
reduce7 写 shared 前先做一轮 WarpReduce 最终版：shared 只存 warp 数个值

一、reduce0：朴素交错寻址 → warp divergence

for (int s = 1; s < blockDim.x; s <<= 1) {
    if ((tid % (s * 2)) == 0) { data[tid] += data[tid + s]; }
    __syncthreads();
}

活跃线程 = 0, 2s, 4s, …，在一个 warp 内活跃 lane 和闲置 lane 交错排列。

⚠️ 问题不是"有线程闲着"（归约本来就会越来越多线程闲）。问题是活跃 / 闲置 lane 在同一个 warp 里交错 → warp 必须把 if 的两个分支串行各跑一遍（SIMT 一次只能走一条路径）→ warp divergence。早期迭代最严重（半数 lane 在每个 warp 里穿插）。

二、reduce1：活跃线程压到前排 → 消 divergence，但 bank conflict

for (int s = 1; s < blockDim.x; s <<= 1) {
    int index = 2 * s * tid;                       // 改：不再用 tid 直接寻址
    if (index < blockDim.x) { data[index] += data[index + s]; }
    __syncthreads();
}

活跃线程变成 tid = 0,1,2,… 连续一段 → 整个 warp 要么全活跃要么全闲，分支不再在 warp 内分裂 → divergence 消失。

⚠️ 代价：index = 2s·tid 让线程以 stride 2s 访问 shared memory。s=1 时 thread 0/1/2… 落在 bank 0/2/4…，thread 0 与 thread 16 撞同一个 bank → 2-way bank conflict；s 越大冲突越重。换句话说：divergence 治好了，bank conflict 冒出来了。

💡 bank conflict 速记：shared memory 分 32 个 bank，warp 内 32 线程同拍访问，若多个线程命中同一 bank 的不同地址，硬件只能串行服务 → 慢。连续访问（线程 i 访问地址 i）天然无冲突。

三、reduce2：顺序寻址 → 消 bank conflict

for (int s = blockDim.x / 2; s > 0; s >>= 1) {     // 改：stride 从大到小
    if (tid < s) { data[tid] += data[tid + s]; }
    __syncthreads();
}

stride 从 blockDim/2 开始逐步折半。活跃线程 tid < s 连续，且每个线程访问 tid 与 tid+s——相邻线程命中相邻 bank → 无 bank conflict，同时无 divergence。

⚠️ 代价：首轮只有前一半线程（tid < blockDim/2）干活，另一半从一开始就闲着。这是下面 reduce3 要榨干的浪费。

四、reduce3：load 时顺手加一次 → 消首轮空闲 + block 数减半

reduce2 首轮半数线程闲——那就让每个线程一次读两个元素、读的时候就加起来，一个 block 因此覆盖 2·blockDim 个元素，load 阶段无人空闲。

int i = blockIdx.x * blockDim.x * 2 + tid;          // 一个 block 管 2*blockDim 个元素
data[tid] = (i < N              ? d_A[i]              : 0.f)
          + (i + blockDim.x < N ? d_A[i + blockDim.x] : 0.f);   // load 顺手加一次
__syncthreads();
// 之后接 reduce2 的顺序寻址归约

💡 全局看：每个 block 干两倍的活 → block 总数减半 → 整个 grid 少一半"读全局 + 多轮 sync 归约"的开销。这是 reduce6 grid-stride（一个 thread 扛 n 个元素）的雏形。

五、reduce4：最后一个 warp 用 warp shuffle → 去 sync + 去 shared

这是从"shared memory 归约"跨到"warp 原语归约"的关键一跃。

核心观察：当 s ≤ 32，所有活跃线程都落在 warp 0 内部。warp 是 SIMT 锁步执行的——这几轮根本不需要 __syncthreads，也不必走 shared memory。直接用 warp shuffle 让 lane 之间寄存器对寄存器传值。

5.1 四个 shuffle 函数（对着图）

__shfl_*_sync(mask, var, arg, width)：每个 lane 拿自己的 var，按规则去读别的 lane 的 var，结果落在自己的寄存器里。

__shfl_sync(_, x, srcLane, w)——所有 lane 读同一个 srcLane（广播）：

width 把 warp 切成若干独立小组，各组各自广播：

__shfl_up_sync(_, x, d, w)——lane i 读 i−d（向高位搬，前 d 个 lane 保留自己），用于 prefix scan：

__shfl_down_sync(_, x, d, w)——lane i 读 i+d（向低位搬），reduce 用它：

__shfl_xor_sync(_, x, m, w)——lane i 与 i^m 互换（蝶形），all-reduce 用它：

5.2 down vs xor 做 WarpReduce

down = 漏斗（结果只在 lane 0）：

val += __shfl_down_sync(_, val, 16/8/4/2/1)，每轮把后半折叠加到前半。5 轮（满 warp）后只有 lane 0 拿到完整和，其余 lane 是中间值。reduce 只要一个总和、由 tid==0 写出，所以用 down 足够。

xor = 蝶形（每个 lane 都有结果）：

val += __shfl_xor_sync(_, val, 16/8/4/2/1)，每轮和异或伙伴互换并相加。5 轮后每个 lane 都拿到完整和（这就是 all-reduce）。

💡 怎么选（用真 kernel 记）：

  • 纯 sum reduce → down：结果只写一次。
  • RMSNorm / softmax → 必须 all-reduce（xor）：算出 Σx² / Σexp 后，每个线程都要拿它去归一化自己负责的那几维，不是只有 lane 0 用 → 和必须回到所有 lane。

一句话：结果只写一次 → down；每个线程都要拿结果继续算 → xor。

（图里画的是 8-lane 示例所以只有 4→2→1 三步；满 warp 是 16→8→4→2→1 五步。）

5.3 代码：改前 → 改后

// 改前（reduce2/3）：最后 6 轮还在 shared memory 里跑，每轮都 sync
for (int s = blockDim.x/2; s > 0; s >>= 1) {
    if (tid < s) data[tid] += data[tid + s];
    __syncthreads();
}

// 改后（reduce4）：归约到剩 64 个就交给 warp shuffle，无 sync、无 shared（设 blockDim ≥ 64）
for (int s = blockDim.x/2; s > 32; s >>= 1) {
    if (tid < s) data[tid] += data[tid + s];
    __syncthreads();
}
float sum = (tid < 32) ? data[tid] + data[tid + 32] : 0.f;   // 64→32 折进寄存器
if (tid < 32) sum = warpReduce<32>(sum);                      // 32→1 全 shuffle，无 sync
if (tid == 0) d_A[blockIdx.x] = sum;
// WarpReduce（down 版，模板 → 完全展开）
template<int warpSize>
__device__ __forceinline__ float warpReduce(float sum) {
    if (warpSize >= 32) sum += __shfl_down_sync(0xffffffff, sum, 16);
    if (warpSize >= 16) sum += __shfl_down_sync(0xffffffff, sum, 8);
    if (warpSize >= 8)  sum += __shfl_down_sync(0xffffffff, sum, 4);
    if (warpSize >= 4)  sum += __shfl_down_sync(0xffffffff, sum, 2);
    if (warpSize >= 2)  sum += __shfl_down_sync(0xffffffff, sum, 1);
    return sum;
}

⚠️ mask 不是摆设（面试常踩）：0xffffffff 表示"这 32 个 lane 都参与"。shuffle 要求被读的 lane 此刻必须活跃且在 mask 里，否则读到未定义值。这正是 reduce7 里 sum = tid < NUM_WARPS ? data[tid] : 0.f 要补 0 的原因——warp 0 做最后归约时，超过 NUM_WARPS 的 lane 在 shared 里没有有效数据，得先用 0 垫掉再 reduce。

六、reduce5：模板完全展开

把 blockSize 做成模板参数，循环边界变成编译期常量 → 编译器把 for 循环和 if (blockSize >= N) 全部展开成无分支直线代码，去掉循环计数、条件判断的运行时开销。

template<int blockSize>
__global__ void reduce5(...) {
    // ... load ...
    if (blockSize >= 512) { if (tid < 256) data[tid] += data[tid+256]; __syncthreads(); }
    if (blockSize >= 256) { if (tid < 128) data[tid] += data[tid+128]; __syncthreads(); }
    if (blockSize >= 128) { if (tid <  64) data[tid] += data[tid+ 64]; __syncthreads(); }
    float sum = (tid < 32) ? data[tid] + data[tid + 32] : 0.f;
    if (tid < 32) sum = warpReduce<32>(sum);
    if (tid == 0) d_A[blockIdx.x] = sum;
}

💡 这就回答了"完全展开在不在最终版"：在。但它不再是教科书里"展开 shared memory 归约树"——因为最后一个 warp 已经被 warp shuffle 接管，warpReduce<> 模板本身就是全展开的；reduce5 展开的只剩 blockDim/2 → 64 这几层 shared 归约。

七、reduce6：grid-stride loop（网格跨步循环）

一句话：让一个线程负责多个元素，跨步遍历整个数组，先在寄存器里把这些元素累加好，再进 shared 归约。

float sum = 0.f;
for (int i = blockIdx.x*blockDim.x + tid; i < N; i += gridDim.x*blockDim.x)
    sum += d_A[i];        // 寄存器累加，步长 = grid 总线程数
// 之后 sum 进 shared / warp 归约

为什么步长是 gridDim.x*blockDim.x（grid 总线程数）？因为这样同拍里相邻线程仍访问相邻地址 → 全局访存保持 coalesced；下一拍整个 grid 一起往后跳一整段。

三个好处：

  1. grid 大小与 N 解耦：grid 不再按 N 开（N 一亿就开一亿/blockDim 个 block），而是按 GPU 占用率开（如 #SM × 每 SM 可驻留 block 数）。block 少 → 最后跨 block 归约的中间结果少。
  2. 拿廉价的寄存器加法换昂贵的 shared 归约层级：一个线程把 N / 总线程数 个元素先在寄存器里加掉，省去多层 shared memory 归约树（算法层面的 cascading）。
  3. 访存连续对齐，带宽利用率高。

📌 图片占位

内容：横轴是整个 global 数组，画 thread 0 / 1 / 2… 各自用箭头跨步命中第 i、i+T、i+2T… 个元素（T = grid 总线程数），体现"相邻线程相邻地址、整体跨步前进"。
一句话看图结论：固定大小的 grid 用跨步循环吃下任意长度的数组。

八、向量化访存（vectorized load）

让一个线程一条指令读 4 个连续 float（float4），而不是 4 条指令读 4 次：

float4 v = reinterpret_cast<const float4*>(d_A)[idx];
sum += v.x + v.y + v.z + v.w;

收益：访存指令数 ÷4，单条指令搬运的字节更多 → 更接近打满显存带宽。和 grid-stride 叠加用（每次跨步读一个 float4）。

⚠️ 前提：地址要 16 字节对齐，元素数最好是 4 的倍数（否则尾部要标量兜底）。

九、reduce7：最终版 = 写 shared 前先 WarpReduce

把 “BlockReduce” 拆成两层 WarpReduce，shared memory 退化成只存"每个 warp 一个部分和"。

流程（对应你贴的代码）：

① grid-stride：每个线程把负责的多个元素累加进寄存器 sum
② 每个 warp 各自 WarpReduce → 每个 warp 出 1 个部分和（纯 shuffle，无 sync）
③ 各 warp 的部分和按 warpId 写进 shared（只写 NUM_WARPS 个值）
④ __syncthreads（全程仅此一次）
⑤ warp 0 读回这 NUM_WARPS 个值（不足补 0）→ 再 WarpReduce<NUM_WARPS> → block 总和
⑥ tid==0 写出
template<int blockSize>
__global__ void reduce7(float* d_A, const int N) {
    extern __shared__ float data[];
    int tid = threadIdx.x;
    int i   = blockSize * blockIdx.x + tid;

    // ① grid-stride 累加进寄存器
    float sum = 0.f;
    for (int index = i; index < N; index += blockSize * gridDim.x) sum += d_A[index];

    // ② 每个 warp 先 WarpReduce
    sum = warpReduce<WARP_SIZE>(sum);
    // ③ 按 warpId 写 shared
    if ((tid & (WARP_SIZE - 1)) == 0) data[tid / WARP_SIZE] = sum;

    __syncthreads();    // ④ 仅此一次

    // ⑤ warp 0 归约各 warp 的部分和
    constexpr int NUM_WARPS = CEIL(blockSize, WARP_SIZE);
    if (tid < 32) {
        sum = tid < NUM_WARPS ? data[tid] : 0.f;     // 补 0：见 5.3 的 mask 易错点
        sum = warpReduce<NUM_WARPS>(sum);
    }
    if (tid == 0) d_A[blockIdx.x] = sum;             // ⑥
}

为什么这是终点：shared memory 只承载 NUM_WARPS（如 8）个值、只用一次，彻底没有多层 shared 归约树 → shared 流量和 sync 都压到最低，所有归约都在 warp shuffle（寄存器）里完成。

reduce7 用到的优化清单：grid-stride（①）+ warp shuffle WarpReduce（②⑤）+ 两层 BlockReduce 结构 + 模板完全展开（warpReduce<> / template<blockSize>）。唯独没显式做向量化访存——这是它还能再叠加的一招（把①的标量 load 换成 float4）。

十、面试要写到哪种程度？

结论：至少写到 warp shuffle 版（reduce4 级别），这是现在的"及格线 / 加分项"分界。

  • reduce0→2（divergence / bank conflict / 顺序寻址）= 必答：这三个问题和对应改法是 reduce 题的基本功，讲不出等于没准备。
  • WarpReduce（__shfl_down_sync）= 期望终点：能手写 warpReduce 说明你懂 warp 锁步、warp 原语、寄存器级归约——这是面试官想听的"现代写法"。多个面经提到"至少到 warpReduce"就是这个意思。
  • reduce3（load 先加）/ reduce6（grid-stride）= 嘴上能说就加分：体现你有"算法层面减少归约量"的意识。
  • reduce5（完全展开）/ float4 向量化 = 锦上添花：时间够再写。

实操建议：手撕时先写干净的 reduce2（顺序寻址）做骨架，再说一句"最后一个 warp 我换 warp shuffle 省掉 sync"，把 warpReduce 写出来。顺序寻址 + WarpReduce 这个组合 = 甜点：正确、够快、把关键知识点都展示了。若被追问"线上怎么写"，口述 reduce7（grid-stride + 两层 warp reduce）即可。

易错点速查 List

# 易错点 正确理解
1 reduce0 的问题是"有线程闲着"？ 不是。归约本就越来越多线程闲。问题是活跃 / 闲置 lane 在 warp 内交错 → 分支串行 → warp divergence。
2 reduce1 把活跃线程压前，代价是什么？ divergence 消失，但 index=2s·tid 造成 stride 访问 → bank conflict。治好一个冒出另一个。
3 reduce2 顺序寻址就完美了？ 无 conflict 无 divergence，但首轮半数线程空闲（reduce3 来榨干）。
4 reduce4 为什么最后一个 warp 能去掉 __syncthreads？ 因为 s≤32 时活跃线程全在一个 warp 内，warp 锁步执行天然同步，且 shuffle 走寄存器、不碰 shared。
5 down 和 xor 的 WarpReduce 一样？ 不一样。down=漏斗，结果只在 lane 0；xor=蝶形，每个 lane 都有结果（all-reduce）。
6 reduce 该用 down 还是 xor？ 纯 sum 用 down（写一次）；RMSNorm / softmax 用 xor（每个线程都要拿和归一化自己那几维）。
7 shuffle 的 mask（0xffffffff）可有可无？ 不可。被读 lane 必须活跃且在 mask 内，否则未定义值。这就是 reduce7 要补 0 的原因。
8 "完全展开"在最终版里没有了？ 有。warpReduce<> 模板和 template<blockSize> 就是编译期全展开；只是展开对象从 shared 归约树变成 warp shuffle 序列。
9 grid-stride 的步长为什么是 grid 总线程数？ 这样同拍相邻线程仍访问相邻地址 → 保持 coalesced；整体每拍跳一整段。它让 grid 大小与 N 解耦，用寄存器加法换 shared 归约层级。
10 reduce7 的 shared memory 还存整个 block 的数据？ 不。只存 NUM_WARPS 个 warp 部分和、只用一次，没有多层 shared 归约树。
11 手撕 reduce 写到 reduce2 就够？ 不够。及格线是 WarpReduce（reduce4）；顺序寻址骨架 + warpReduce 是最佳作答组合。
