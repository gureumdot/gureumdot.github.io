【挑战四 QA 梳理：CUDA Graph】
以下围绕「cuda graph 无法开启 piecewise 优化模式」这条经历，梳理面试可能追问的 CUDA Graph 知识点，重点是几个最易理解错的地方。

挑战四 QA·1：CUDA Graph 的真正优化目标（含「为什么要固定 shape」）

一句话结论：直接原因是把多次 kernel 发射合并成一次发射，消除 CPU 侧的 kernel launch 开销。固定 shape 只是前置条件，不是加速来源本身。

它录的是什么：常见误解以为 Graph「把计算大小写好了所以快」。错。Graph 录的是一整串 kernel 的调用拓扑（哪些 kernel、什么顺序、依赖、各自的 launch 参数），不是 tensor 的 shape。

加速从哪来：eager 模式下每个 kernel 都要 CPU 走一遍提交（参数打包 → 驱动校验 → 推进 GPU 队列），每次约 5~10μs。LLM 一个 decode step 有几百个 kernel，单个计算可能只要 2~3μs，于是 GPU 算得比 CPU 喂得快，在「饿肚子」干等（launch-bound）。CUDA Graph 把这几百次提交录制一次，之后每 step 只发一条 cudaGraphLaunch，几百个 kernel 在 GPU 侧按录好的拓扑自动接力，几百次提交压成一次。

和 #pragma unroll 的区别（易混点）：两者唯一共性是「提前确定 → 干掉运行时重复固定开销」这个直觉，但消除的开销类型不同。#pragma unroll 消除循环控制开销（计数器、边界判断、跳转），是单 kernel 内部、GPU 指令级的优化；CUDA Graph 消除 kernel launch 开销（CPU→GPU 提交路径），是跨 kernel、CPU 提交侧的优化。

为什么要固定 shape，动态 shape 会怎样？（这是问题一的一部分）

准确的因果链（注意表述陷阱）：「动态 shape 就必须由 GPU 发射」是错的。动态 shape 只是让你没法用 CUDA Graph，于是退回 CPU 逐个发射（eager）：

shape 动态  →  无法录制 graph  →  只能 eager（CPU 逐个发射）
shape 固定  →  能录制 graph    →  回放时 GPU 接力，CPU 只发一次

「GPU 自己接力」是 graph 录制成功后的结果，不是处理动态 shape 的手段。

为什么动态 shape 录不了：录制 = 把变量烤成常量。Graph 录的是一串参数被烤死的 kernel 节点，每个节点固化三样东西，而这三样都随 shape 变：① grid/block 配置（ceil(M*N/block_size) 这类按 shape 算的常数）；② 显存地址（录制那刻的具体指针）；③ shared memory（SRAM）/ register 预分配。录制时这些被当字面常量写进节点，回放时 GPU 照常量触发，不需要 CPU 临时算任何东西，这才是它能跳过 CPU 的原因。动态 shape 下这些值要运行时才知道、不同 shape 全不一样，必须 CPU 先算好再分配给 GPU → 回到 eager。

加分点：GPU 能自己发射吗？ 问题不在「算」，而在「发射」这个动作传统上只在 CPU 手里。发射 kernel 是一次 host 向 device 提交任务的系统级动作，发起权在 host；GPU 是被动执行器，即便算出 grid 大小，结果还得传回 CPU 发射，绕一圈 launch 开销原封不动。但严格说这是传统 host-launch 模型的限制，不是物理铁律：CUDA 有 Dynamic Parallelism（GPU 上的 kernel 直接再发射子 kernel）、12.4 起有 device-side graph launch。实际 vLLM/SGLang 不用，是工程权衡（device 端发射开销不小、生态全按 host-launch 写、固定 shape 方案够用），非「不可能」。面试若追问「GPU 真没办法吗」，答出这个即高质量回答。

挑战四 QA·2：为什么 piecewise 之前 decode 能做 CUDA Graph、prefill 不能？

先纠一个易错点：「decode 是固定 shape」是不准确的。准确说法是——decode 之所以能做 graph，不是天生 shape 固定，而是它的动态性可以被治理成固定；prefill 的动态性治不了。

  • prefill：输入是整段 prompt（50 / 800 / 2000… token），seqlen 直接进 shape，且取值连续无法枚举，抓 graph 抓不完 → 天然动态。
  • decode 有三个维度，逐个被治理：① 输入 token 数恒为 1（天然固定）；② batch size 会变（请求进出），但可以预抓一批固定 batch size 的 graph，运行时 padding 到最近一档复用；③ KV cache 长度每步涨、每条不同，但它藏在 attention kernel 内部（通过指针/长度元数据传入），不进外层 tensor 的 shape，所以外层算子的 graph 不受它影响。

最关键认知：从外层算子（RMSNorm、QKV proj、MLP）视角，decode 的 shape 是 [batch, 1, hidden]，batch 固定它们就全固定，KV 长度根本不出现。KV 长度只在 attention 内部起作用 —— 这恰恰就是 piecewise 的动机。

挑战四 QA·3：piecewise 到底解决什么问题，为什么attention没办法做cudagraph

一句话：piecewise 建立在 chunked prefill 基础上，做attention分离操作，非attention可以做cuda graph，attention正常走eager

完整逻辑链（这条最重要）：

  1. prefill 做不了 CUDA Graph 的根因：一个 batch 里所有请求的 token 总量不确定（seqlen 连续变化、无法枚举）。
  2. chunked prefill 解决「长度不确定」：可以让长度等长
  3. 但 attention 这里用不了 chunk：因为attention要计算所有token之间的注意力，不能切割分开计算，要不然后面的chunk块就看不懂前面chunk块的注意力
  4. 所以 piecewise 必须把 attention 这一段从 graph 里去除掉
  • 挑战五：MoE 架构模型无法通过 cudagraph 抓取
    • 原因：抓取过程中不允许有异构计算，但 moe_forward_native 中计算每个 tokens_per_expert 会移步到 CPU 进行计算
    • 解决方案：保证全部在 GPU 上执行（可参考 NPU 的做法）；重新设计 MoE 计算逻辑，针对性设计 MoE 专家计算逻辑：1. 专家初始化：根据 token 与专家选择；2. 专家并行计算
挑战五 QA：MoE + CUDA Graph 的异构计算冲突

待补充：为什么 tokens_per_expert 这一步必须在 CPU 上算（是因为 dynamic indexing 吗）、CUDA Graph 不允许异构计算的本质（=不允许 device-host 同步点）、NPU 的做法是什么、重新设计的 MoE forward 怎么把 expert dispatch 留在 GPU 上等。

2. 基于 ONNX 计算图的 YOLOv11 架构微调、算子调度优化与图重构

    • 现象：计算图中经过所有激活层的卷积都在复用第一层的卷积对象
    • 原因：架构内部所有激活层共用了同一个卷积对象，未做区分
    • 修改：/root/qdrop/qdrop/model/code/yolo11/ultralytics/ultralytics/nn/modules/conv.py，每次重新初始化卷积对象，而不是反复调用同一个对象
挑战一 QA：卷积对象复用为什么会破坏 PTQ

待补充：复用对象会让 PTQ 的 calibration（per-layer min/max 采样）拿到的统计量被多层混合污染、QDrop 这类按层独立量化的算法为什么必须每层有独立对象、修改后是否会因为不再复用而显著增大显存/参数量。

  • 挑战二：算子封装（图重构）
    • 现象：部分计算无法直接生成完整计算图
    • 原因：某些计算逻辑太复杂，ONNX tracer 会深入到内部细节，导致图不完整
    • 解决：将相关函数设置为算子类型（op），tracer 不再展开其内部实现，从而生成完整计算图
    • 修改：/root/qdrop/mqbench/mqbench/advanced_ptq.py
挑战二 QA：torch.onnx.export 的 tracer 展开问题

待补充：tracer 为什么会"深入到内部细节"导致图不完整（控制流/Python 原生 op 的副作用）、@torch.fx.wrap / custom symbolic 注册的作用、把函数封装成 op 后 ONNX 里是怎么的节点形态、对后续 mqbench 量化的影响。

  • 挑战三：box 回归值始终为 0（图重构）
    • 现象：YOLOv11 推理过程中，回归值（box 坐标）计算结果始终为 0
    • 原因：封装 box 计算算子时，内部每次调用都会重新初始化成员变量，导致计算始终从初始值（0）开始，而不是使用正确的成员变量
    • 解决：直接读取成员变量参与计算，去掉每次调用时的重复初始化
挑战三 QA：DFL/box decode 的状态保留问题

待补充：YOLOv11 box decode 用的 DFL（Distribution Focal Loss）需要哪些常量张量（projection vector / anchors / stride）、为什么封装成 op 后 __init__ 会被反复调用、修改方案是改成 buffer 还是直接外部传入。

3. 多芯异构调度策略 & C++ 重构模型部署链路

  • 核心优化：NMS 从 GPU 迁移到 CPU
    • 分析：对整个推理流程进行 profiling，发现 NMS（Non-Maximum Suppression，去除重复框）环节的特点是数据量小、逻辑串行、不适合大规模并行
    • 决策：NMS 处理过程改在 CPU 上执行，GPU 专注于卷积/矩阵等并行计算密集型部分
    • 效果：减少了不必要的 GPU-CPU 数据搬运开销，整体推理链路更高效
NMS 异构调度 QA：为什么 NMS 反而放回 CPU 更快

待补充：NMS 的串行依赖本质（IoU 比较 + 贪心抑制）、GPU 上做 NMS 的代价（kernel launch 开销 + 同步点 + 小数据量利用率低）、CPU/GPU 数据传输如何避免（结果集小、和后处理同步 pipeline）、和"GPU 饿肚子"那部分 CUDA Graph 知识的呼应。

C++ 重构部署链路 QA：从 Python 推理 demo 到 C++ 部署的工程要点

待补充：原 Python 推理链路的瓶颈在哪（GIL/数据拷贝/解释器开销）、C++ 侧用的推理引擎（自研 / ONNX Runtime / TensorRT-LLM-like）、内存池/线程池设计、和 NMS-on-CPU 异构调度怎么协同（CPU 线程池接 GPU stream 的 callback）。
