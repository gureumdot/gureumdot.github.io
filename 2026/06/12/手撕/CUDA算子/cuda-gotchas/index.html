CUDA 算子易错点 / 易混淆点汇总

CUDA GEMM 算子优化 易错点
手撕 CUDA算子
Publish Date:   2026-06-12

持续更新。专收那些「乍看懂了、面试一追问就露馅」的点。每条配图。

易错点 1：内存事务 vs 访存量（两套正交的度量）

一句话：内存事务不改变「访存量」（计算访存比的分母），但决定搬运效率；访存量靠 warp 复用 / 广播来压低，从而抬高计算访存比。这是两条正交的优化线，缺一不可。

1.1 先分清「访存量」到底指哪个

「访存量」这个词在两个层级上含义完全不同，混了就全乱：

同一次 warp 读 tileB,两种数法

逻辑读出量（算法层）
warp 要喂给 ALU 多少个独立 float
16 个不同 col → 算 16
这就是访存比的分母
只跟「算法怎么复用」有关

真实搬运量（硬件层）
DRAM/L2 实际搬了多少字节
16 个连续 → 合并成 1 次事务
这是「搬得快不快」
只跟「访问连不连续」有关
你把「访存量」理解成右边的事务/字节,所以才觉得「都是 1 次,怎么会变」——其实分母数的是左边

  • 左边（逻辑读出量）：warp 这一拍逻辑上要读出多少个不同的 float 值喂给计算单元。这才是计算访存比的分母。 它由算法决定，跟硬件用几次事务搬过来无关。
  • 右边（真实搬运量）：硬件实际从 DRAM/L2 搬了多少字节。这由访问是否连续、对齐决定，对应内存事务 / 合并（coalescing）。

⚠️ 16 个连续 float 物理上确实合并成 1 次事务，你这个直觉没错。但「合并成几次事务」是搬运打包问题，不改变「逻辑上读了 16 个值」这件事。分母数的是逻辑值的个数，不是事务数。

1.2 事务数影响什么：同样 32 个 float，1 次 vs 2 次事务

事务不改访存比的分母，但直接决定耗时和带宽利用率：

读同样 32 个 float = 128 字节，两种打包

连续 → 1 次事务
恰好凑满 1 个 128B 内存块
真实搬运：128 字节
耗时：1 拍
带宽利用率：100%

分散 → 2 次事务
跨 2 个 128B 块（跨界 / stride）
真实搬运：256 字节（浪费一半）
耗时：2 拍
带宽利用率：50%

事务有最小粒度（典型 128B 一块）。32 个 float 散落在 2 个块里，硬件必须发 2 次事务、每次都搬整块 128B，搬 256B 才拿到要的 128B，多搬的一半纯浪费，耗时翻倍。

1.3 在一次事务内，用 warp 优化访存量

真正减少逻辑读出量的机制是 shared memory 的广播——同一 warp 内多个线程读同一地址只算一次。它发生在数据复用的方向上。GEMM 里 warp 排布形状决定了广播省在哪一维：

warp 形状 tileB 读出（沿 x，Wx） tileA 读出（沿 y，Wy） 分母 Wx+Wy 计算访存比 = 128/(Wx+Wy)
16×2 16 2 18 7.1
8×4 8 4 12 10.7 ✅

分子（算力）固定：32 线程 × 64 乘加 × 2，与 warp 形状无关。变的只有分母。Wx×Wy=32 约束下，和最小 → 越接近正方形越优（和定积、积定和），所以 8×4 / 4×8 最佳。这条「正方形最优」原理一路向上同样适用于 thread tile（Tm×Tn 选 8×8）和 block tile。

1.4 两套度量正交：哪个优化手段服务于哪一头

这张表是分清两套度量的试金石，也是面试高频追问点：

优化手段 服务于 原理
tiling（block/warp/thread tile） 访存比（压分母） 把数据搬进 shared/寄存器后反复复用
广播（broadcast） 访存比（压分母） 同 warp 读同地址只算一次
FLOAT4 向量化读取（LDS.128） 事务（提带宽利用率） 一条指令搬 4 个连续 float，压缩访问次数
As 转置存储（As[k][m]） 事务（让访问连续） 外积时 A 的列变连续，才能 float4 合并读
内存合并（coalescing） 事务（提带宽利用率） 让 warp 访问落在连续地址

🎯 超高频坑：「As 为什么要转置存储？」——转置是为了让该读的 4 个 float 在地址上连续，从而合并成一次 128 位事务，纯事务层面的理由。很多人误以为「转置是为了减少读取量 / 优化访存比」——错，转置一个 float 都不会少读，逻辑读出量纹丝不动。能不能分清这点，就看你有没有真把两套度量拆开。

1.5 coalescing / broadcast / bank conflict 辨析

三个概念极易混为一谈，但分属不同层级：

概念 发生层级 触发条件 解决什么
coalescing（合并） global → L2/DRAM 同 warp 访问连续对齐地址 减少事务数、提带宽利用率
broadcast（广播） shared memory 同 warp 多线程读同一地址 降低逻辑读出量（省一次读）
bank conflict（冲突） shared memory 同 warp 多线程读同 bank 不同地址 避免则单次完成，否则串行

⚠️ 易记反：广播只在 shared 且 同一 warp 内成立；不同 warp 读同地址不广播，global memory 也没有广播（global 那层叫 coalescing）。broadcast 与 bank conflict 是一对：同 bank 同地址 = 广播不冲突；同 bank 不同地址 = 冲突串行。

易错点 2：计算访存比 vs SM 占有率

一句话：访存比定上限（天花板有多高），占有率定能否爬到上限。两个都要高，缺一不可。

两个指标,回答两个不同的问题

计算访存比（算术强度）
FLOP / Byte，算法本身决定
回答：这活儿最快能多快
→ 决定性能「上限 / 天花板」
低 = 卡带宽，堆线程也没用
靠 tiling / 复用 / 广播 抬升

SM 占有率
活跃 warp 数 / 硬件上限
回答：能不能跑满那个上限
→ 决定「能否逼近天花板」
太低 = warp 不够掩盖延迟
靠控制寄存器/shared 用量抬升
为什么两个都要高（缺一不可）

访存比高 + 占有率低
天花板很高，却够不着
warp 太少，延迟掩盖不住
计算单元大量空转
✗ 不行

占有率高 + 访存比低
warp 塞满，可天花板很低
全卡在搬数据上
再多 warp 也突破不了带宽墙
✗ 不行

  • 访存比（算术强度）决定天花板有多高：roofline 模型里那条斜线 + 平台线，FLOP/Byte 越高，理论峰值越高。
  • 占有率决定能不能爬到天花板：它提供足够的活跃 warp 来掩盖访存 / 指令延迟，把硬件喂饱。占有率低 → 延迟掩盖不住 → 实际性能远低于天花板。

✅ 准确口诀：访存比定上限，占有率定能否达到上限。

🧭 直觉对照（防止自己记反）：我最初记成「占有率定下限」。这个直觉可以保留，但要理解对——它不是说占有率给性能设了个保底地板（占有率高也不保证性能好，见右下角「占有率高+访存比低」照样慢）。正确含义是：占有率太低会把实际性能从天花板拖到很惨的位置。占有率更像「达成率 / 利用率」，不是「保底线」。
